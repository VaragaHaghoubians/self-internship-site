12-Month Capstone Project Roadmap

This file defines the deliverable projects for Months 5–9.
Months 1–4 taught the skills. Months 5–9 are where you prove them.
Each capstone has:

- Skills demonstrated (what the project proves to an employer)

Capstone 1 — Month 5 (Weeks 17–20): Django Smart Home Portal

Elevator pitch: "I upgraded my Flask IoT server to a production Django app with user accounts, a PostgreSQL database, and deployed it to AWS so it's accessible from any phone."

What to Build

Take your Month 4 smart home system (ESP32 → Flask → SQLite → dashboard) and upgrade it:

Month 4 (starter)Month 5 (capstone)
FlaskDjango
SQLitePostgreSQL
No authUser login/logout (Django auth)
localhost onlyLive on AWS EC2 (public URL)
Manual refreshAuto-refreshing Plotly chart

Technical Spec

django-smart-home/
├── manage.py
├── requirements.txt           ← django, psycopg2, plotly, gunicorn
├── smart_home/
│   ├── settings.py            ← DATABASE = postgresql, SECRET_KEY from env var
│   ├── urls.py
│   └── wsgi.py
├── sensors/
│   ├── models.py              ← SensorReading(device_id, temperature, humidity, timestamp)
│   ├── views.py               ← POST /api/upload/, GET /dashboard/
│   ├── serializers.py
│   └── templates/
│       └── dashboard.html     ← Plotly chart, last reading, anomaly flag
└── Dockerfile

Key tasks in order:

  1. django-admin startproject smart_home + python manage.py startapp sensors
  2. Define SensorReading model, run migrations
  3. Create a POST API endpoint that accepts {device_id, temperature, humidity} JSON
  4. Update your ESP32 Arduino sketch to POST to the new endpoint (change the URL)
  5. Build the dashboard view: query last 200 readings, render Plotly line chart in template
  6. Add Django's built-in LoginRequiredMixin so only logged-in users see the dashboard
  7. Dockerize: Dockerfile + docker-compose.yml (Django + PostgreSQL containers)
  8. Deploy to AWS EC2: launch t2.micro (free tier), SSH in, docker compose up -d

Acceptance Criteria

  • [ ] ESP32 sends real data every 60 seconds; it appears on the dashboard within 2 minutes
  • [ ] Accessing /dashboard/ without login redirects to /login/
  • [ ] docker compose up starts the full system with one command
  • [ ] Dashboard is accessible at http://<your-ec2-public-ip>/dashboard/ from your phone
  • [ ] GitHub README includes: architecture diagram, setup instructions, screenshot

Skills Demonstrated

Django ORM, REST APIs, user authentication, Docker Compose, AWS EC2 deployment, PostgreSQL.

Capstone 2 — Month 6 (Weeks 21–24): Predictive Maintenance MVP

Elevator pitch: "I built an anomaly detector that watches live sensor data from my ESP32 and sends an email alert when it predicts a cooling failure 2 hours before it happens."

What to Build

Use the time series + ML skills from Month 3 on your real ESP32 data from Month 4.

The Pipeline:

ESP32 sensor (every 60s)
    → Django API (save to PostgreSQL)
    → Background task (Celery or APScheduler) runs every 5 minutes:
        1. Load last 6 hours of readings from DB
        2. Engineer lag features (lag_1h, lag_3h, rolling_1h_std)
        3. Score with pre-trained RandomForestClassifier
        4. If failure_probability > 0.7: send email alert via smtplib
    → Dashboard shows: live reading + failure probability gauge + alert history

Technical Spec

Model training script (train_anomaly_model.py):

# Simulate training data: inject artificial "failure" events into historical readings
# Failure signature: temperature rising > 0.5°C per reading for 3+ consecutive readings
# Label those as anomaly=1, everything else as anomaly=0
# Features: lag_1, lag_3, lag_6, rolling_30min_mean, rolling_30min_std, hour_of_day
# Model: RandomForestClassifier(class_weight='balanced')  ← handles imbalance
# Save with: import joblib; joblib.dump(model, 'anomaly_model.pkl')

Scoring endpoint (sensors/tasks.py):

import joblib, smtplib
from email.message import EmailMessage

model = joblib.load("anomaly_model.pkl")

def score_latest_readings():
    # called every 5 minutes by APScheduler
    df = load_last_6h_from_db()
    features = engineer_features(df)
    prob = model.predict_proba(features.tail(1))[0][1]
    if prob > 0.70:
        send_alert_email(prob)

Libraries: scikit-learn, joblib, APScheduler, smtplib (built-in)

Acceptance Criteria

  • [ ] train_anomaly_model.py runs end-to-end and saves anomaly_model.pkl
  • [ ] Dashboard shows a "failure risk" percentage that updates every 5 minutes
  • [ ] Manually injecting a high temperature (hold your hand on the sensor) triggers an alert email within 10 minutes
  • [ ] GitHub README documents: model accuracy on held-out test set, false positive rate, alert threshold rationale

Skills Demonstrated

Feature engineering on time series, classification model deployment, background scheduling, email alerting, joblib model serialization.

Capstone 3 — Month 7 (Weeks 25–28): Supply Chain Demand Forecasting

Elevator pitch: "I built a demand forecasting system on a Kaggle retail dataset comparing three model classes — statistical, ML, and deep learning — and deployed the best one as a Plotly dashboard."

What to Build

Download the Kaggle Store Item Demand Forecasting dataset (10 stores × 50 items = 500 time series, 5 years of daily sales). Build a forecasting pipeline that:

  1. Trains three models on each time series
  2. Picks the best model per series based on validation MAE
  3. Displays forecasts in a Plotly dashboard with confidence intervals

Technical Spec

Repository structure:

├── notebooks/
│   └── 01_eda.ipynb            ← exploratory analysis (required deliverable)
├── src/
│   ├── features.py             ← create_lag_features(), add_calendar_features()
│   ├── models.py               ← SARIMAModel, ProphetModel, XGBoostModel (classes)
│   ├── evaluate.py             ← mae(), rmse(), mape() functions + comparison table
│   └── dashboard.py            ← Plotly Dash app: store selector + forecast chart
├── tests/
│   └── test_features.py        ← pytest: verify lag features are correctly shifted
├── requirements.txt
└── README.md

Model comparison format (print this at the end of main.py):

Store 1, Item 1 — Validation MAE (last 90 days):
  SARIMA(1,1,1)(1,1,1,7):   12.4 units
  Prophet (weekly+yearly):  10.8 units   ← WINNER
  XGBoost (lag features):   11.3 units

Minimum viable deliverable (focus here first):

  • One store, one item, all three models, comparison table → 1 week
  • Scale to all 50 items in one store, visualize best model per item → 1 week
  • Plotly Dash dashboard with store/item dropdowns → 1 week
  • Deploy dashboard to AWS or Render.com → 1 week

Acceptance Criteria

  • [ ] notebooks/01_eda.ipynb has: time plots, seasonal decomposition, ACF/PACF, summary statistics
  • [ ] Three models are implemented as classes with .fit(train) and .predict(horizon) interface
  • [ ] Comparison table shows MAE, RMSE, MAPE for each model on the same held-out test set
  • [ ] Dashboard lets user select any store + item and see the forecast with confidence interval
  • [ ] tests/test_features.py passes: verify lag feature shift, verify no data leakage

Skills Demonstrated

Time series forecasting (SARIMA, Prophet, XGBoost), model comparison methodology, Plotly Dash, software engineering (classes, tests), cloud deployment.

Capstone 5 — Month 9 (Weeks 33–36): Grand Capstone — Cold Chain Intelligence System

Elevator pitch: "I built a full-stack system that monitors cold storage sensors in real time, predicts cooling failures using machine learning, and automatically computes the optimal inventory transfer plan to minimize spoilage — all in one live dashboard."

What to Build

This is the integration of everything. Every previous capstone feeds into this one:

[Hardware]   ESP32 with temperature sensor in a small cooler box
    ↓
[Ingest]     Django API (Capstone 1) — saves readings to PostgreSQL
    ↓
[Detection]  ML anomaly scorer (Capstone 2) — runs every 5 minutes
    ↓
[Response]   OR-Tools assignment optimizer (Capstone 4) — triggered when risk > 70%
    ↓
[Dashboard]  Unified Django view showing:
               • Live temperature chart (last 6 hours)
               • Failure probability gauge (0–100%)
               • Current inventory levels per location
               • Optimal transfer plan (which pallets go where)
               • Financial impact: estimated spoilage cost if no action taken

Technical Spec: The Decision Pipeline

# optimizer/pipeline.py

def run_cold_chain_pipeline(device_id: str) -> dict:
    """
    Called by the Celery scoring task every 5 minutes.
    Returns a decision object the dashboard renders.
    """
    # Step 1: Load last 6h of readings for this device
    readings = SensorReading.objects.filter(
        device_id=device_id,
        timestamp__gte=timezone.now() - timedelta(hours=6)
    ).order_by("timestamp")

    # Step 2: Engineer features, score with ML model
    df       = pd.DataFrame(list(readings.values()))
    features = engineer_features(df)
    failure_prob = anomaly_model.predict_proba(features.tail(1))[0][1]

    result = {
        "device_id":              device_id,
        "latest_temp_c":          df["temperature"].iloc[-1],
        "failure_probability":    round(failure_prob, 3),
        "financial_risk_usd":     round(failure_prob * AVG_INVENTORY_VALUE, 2),
        "action_required":        failure_prob > ALERT_THRESHOLD,
        "optimal_transfer_plan":  None,
        "alert_sent":             False,
    }

    # Step 3: If high risk, compute optimal inventory transfer
    if result["action_required"]:
        inventory   = fetch_current_inventory(device_id)
        warehouses  = fetch_available_warehouses()
        result["optimal_transfer_plan"] = solve_inventory_assignment(
            inventory, warehouses
        )
        send_alert_email(result)
        result["alert_sent"] = True

    # Step 4: Save decision to DB for audit trail
    Decision.objects.create(**result)
    return result

Interview Story (memorize this)

"For my capstone project, I built a cold chain monitoring system from hardware to deployment.
An ESP32 microcontroller with a temperature sensor sends readings every 60 seconds to a Django API.
A machine learning model runs in the background and scores the latest readings for cooling failure risk.
If the risk exceeds 70%, the system automatically computes the optimal inventory transfer plan
using Google OR-Tools and sends an alert email with the transfer recommendations.
The whole system runs in Docker and is deployed on AWS EC2.
I chose this project because it combines my industrial engineering background — specifically
inventory management and operations research — with data science and software engineering.
It's the kind of system that would be valuable at a pharmaceutical distributor, food logistics company,
or any company managing temperature-sensitive goods."
