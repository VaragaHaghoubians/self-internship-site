#
#  TASK: Master Git practices specific to ML Engineering and AI Engineering roles.
#        This is the most advanced Git level — used at Amazon, Uber, Flexport.
#
#  WHY:  ML projects have unique Git challenges Python/web projects don't:
#          → Large data files that Git can't handle (DVC solves this)
#          → Models that need versioning alongside code
#          → Automated testing of ML pipelines (GitHub Actions)
#          → Pre-commit hooks for formatting and notebook cleanup

cat << 'EOF'

A well-structured ML project that works well with Git:

  my-ml-project/
  ├── .github/
  │   └── workflows/
  │       ├── ci.yml              ← GitHub Actions: run tests on every push
  │       └── deploy.yml          ← GitHub Actions: deploy model on tag
  ├── .gitignore                  ← CRITICAL — never commit data/models/secrets
  ├── .pre-commit-config.yaml     ← pre-commit hooks: format, lint, strip notebooks
  ├── dvc.yaml                    ← DVC pipeline definition
  ├── dvc.lock                    ← DVC lock file (COMMIT THIS — it's like requirements)
  ├── data/
  │   ├── .gitignore              ← ignore all data files
  │   └── .dvc/                   ← DVC pointers (small files — commit these)
  ├── models/
  │   ├── .gitignore              ← ignore .pkl, .joblib, .onnx files
  │   └── model_v1.dvc            ← DVC pointer to model (commit this)
  ├── notebooks/
  │   └── exploration.ipynb       ← should be stripped of outputs before commit
  ├── src/
  │   ├── data_loader.py
  │   ├── features.py
  │   ├── model.py
  │   └── evaluate.py
  ├── tests/
  │   └── test_pipeline.py
  ├── requirements.txt
  └── README.md

KEY RULE: Git tracks CODE. DVC tracks DATA and MODELS.
          Code is small. Data can be gigabytes. Git chokes on large files.

EOF

# ==============================================================================
# LESSON 03 — GIT HOOKS: Automated Code Quality
# ==============================================================================
#
# TEACHER SAYS:
# Git hooks are scripts that run AUTOMATICALLY at key moments:
#   pre-commit  → runs BEFORE a commit completes
#   pre-push    → runs BEFORE a push completes
#   commit-msg  → validates the commit message format
#
# Use case: prevent bad code from EVER entering the repo.
# If the hook fails → commit is BLOCKED until you fix the issue.

echo ""
echo "=== LESSON 03: Git Hooks with pre-commit ==="

cat << 'EOF'

SETUP (one-time):
  pip install pre-commit
  pre-commit install          ← installs hooks into .git/hooks/

CREATE .pre-commit-config.yaml in your project root:
EOF

cat << 'YAML'
# ==========================================
# .pre-commit-config.yaml
# ==========================================
repos:
  # 1. Basic file hygiene:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace      # Remove trailing spaces
      - id: end-of-file-fixer        # Ensure files end with newline
      - id: check-yaml               # Validate YAML files
      - id: check-json               # Validate JSON files
      - id: check-merge-conflict     # Block commits with conflict markers
      - id: detect-private-key       # NEVER commit private keys
      - id: no-commit-to-branch      # Prevent direct commits to main
        args: ['--branch', 'main']

  # 2. Python formatting (Black):
  - repo: https://github.com/psf/black
    rev: 24.1.0
    hooks:
      - id: black
        language_version: python3.11

  # 3. Python imports sorting (isort):
  - repo: https://github.com/PyCQA/isort
    rev: 5.13.2
    hooks:
      - id: isort
        args: ["--profile", "black"]

  # 4. Python linting (flake8 — catches bugs):
  - repo: https://github.com/PyCQA/flake8
    rev: 7.0.0
    hooks:
      - id: flake8
        args: ["--max-line-length=100"]

  # 5. Strip Jupyter notebook outputs (CRITICAL for ML projects):
  - repo: https://github.com/kynan/nbstripout
    rev: 0.7.1
    hooks:
      - id: nbstripout   # Removes cell outputs before commit — keeps notebooks clean
YAML

cat << 'EOF'

USAGE after setup:
  git add .
  git commit -m "feat: add model"

  → pre-commit RUNS AUTOMATICALLY:
    [black] Reformatting model.py...
    [isort] Sorting imports in features.py...
    [nbstripout] Stripping outputs from exploration.ipynb...
    [check-merge-conflict] ✓ passed
    [detect-private-key] ✓ passed

  If any hook FAILS → commit is BLOCKED → fix the issue → commit again

RUN MANUALLY (useful to check before committing):
  pre-commit run --all-files        ← check every file in the repo

BYPASS (emergency only, add a comment explaining why):
  git commit --no-verify -m "emergency: hotfix bypassing hooks"

EOF


# ==============================================================================
# LESSON 04 — GITHUB ACTIONS: CI/CD for ML Projects
# ==============================================================================
#
# TEACHER SAYS:
# CI = Continuous Integration: automatically test code on every push
# CD = Continuous Deployment: automatically deploy when tests pass
#
# GitHub Actions runs YAML files in .github/workflows/
# It starts a fresh virtual machine and runs your commands.
# FREE for public repos. Limited free minutes for private repos.

echo ""
echo "=== LESSON 04: GitHub Actions CI/CD ==="

cat << 'EOF'
Create: .github/workflows/ci.yml
EOF

cat << 'YAML'
# ==========================================
# .github/workflows/ci.yml
# Runs on every push and pull request
# ==========================================
name: CI — Test ML Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main, develop ]

jobs:
  test:
    runs-on: ubuntu-latest       # Fresh Linux VM for each run
    strategy:
      matrix:
        python-version: ["3.11", "3.12"]   # Test on multiple versions!

    steps:
      # 1. Download your code:
      - name: Checkout code
        uses: actions/checkout@v4

      # 2. Install Python:
      - name: Set up Python ${{ matrix.python-version }}
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}

      # 3. Cache pip packages (speeds up future runs):
      - name: Cache pip packages
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}

      # 4. Install dependencies:
      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
          pip install pytest pytest-cov

      # 5. Run pre-commit checks (same hooks as local):
      - name: Run pre-commit hooks
        uses: pre-commit/action@v3.0.1

      # 6. Run tests with coverage:
      - name: Run tests
        run: |
          pytest tests/ -v --cov=src --cov-report=xml

      # 7. Fail if coverage drops below 80%:
      - name: Check coverage threshold
        run: |
          pip install coverage
          coverage report --fail-under=80

      # 8. Upload coverage report (visible in GitHub PR):
      - name: Upload coverage report
        uses: codecov/codecov-action@v4
        with:
          file: coverage.xml
YAML

# ==============================================================================
# LESSON 05 — DVC: Data Version Control
# ==============================================================================
#
# TEACHER SAYS:
# Git can't handle large files (CSVs, model weights, datasets — GBs of data).
# DVC (Data Version Control) solves this:
#   → Git tracks CODE and DVC POINTER FILES (.dvc)
#   → DVC stores the ACTUAL large files in remote storage (S3, Google Drive, etc.)
#   → Together: you can reproduce ANY experiment by checking out code + pulling data

echo ""
echo "=== LESSON 05: DVC — Data Version Control ==="

cat << 'EOF'

INSTALL:
  pip install dvc
  pip install dvc-s3      ← if using Amazon S3 as remote storage
  pip install dvc-gdrive  ← if using Google Drive

INITIALIZE (inside your Git repo):
  git init                        # (if not already)
  dvc init                        # Creates .dvc/ directory
  git add .dvc/
  git commit -m "chore: initialize DVC"

CONFIGURE REMOTE STORAGE (where DVC uploads your data):
  # Google Drive (free, easy to start):
  dvc remote add myremote gdrive://YOUR_FOLDER_ID
  dvc remote default myremote
  git add .dvc/config
  git commit -m "chore: configure DVC remote"

  # Amazon S3 (for production):
  dvc remote add s3remote s3://my-bucket/dvc-data
  dvc remote default s3remote

TRACK LARGE FILES:
  dvc add data/train.csv          # DVC tracks it, creates data/train.csv.dvc
  git add data/train.csv.dvc data/.gitignore
  git commit -m "data: add training dataset v1"
  dvc push                        # Upload actual CSV to remote storage

  dvc add models/churn_model.pkl  # Same for model files
  git add models/churn_model.pkl.dvc
  git commit -m "model: save trained churn model v1"
  dvc push

REPRODUCE ON ANOTHER MACHINE (or after git clone):
  git clone <your-repo>
  pip install dvc
  dvc pull                        # Download all tracked data + models from remote

THE BEAUTIFUL RESULT:
  git checkout v1.0.0             # Code at v1.0.0
  dvc pull                        # Data and model that match that code version
  → Perfectly reproducible experiment!

DVC PIPELINES (automate training):
  # Create dvc.yaml to define your pipeline stages:
EOF

cat << 'YAML'
# dvc.yaml — ML pipeline definition
stages:
  preprocess:
    cmd: python src/preprocess.py
    deps:
      - src/preprocess.py
      - data/raw/customers.csv       # if this changes → re-run stage
    outs:
      - data/processed/features.csv

  train:
    cmd: python src/train.py
    deps:
      - src/train.py
      - data/processed/features.csv
    outs:
      - models/churn_model.pkl
    metrics:
      - metrics/train_metrics.json   # DVC tracks these metrics!

  evaluate:
    cmd: python src/evaluate.py
    deps:
      - src/evaluate.py
      - models/churn_model.pkl
    metrics:
      - metrics/eval_metrics.json
YAML

cat << 'EOF'

RUN THE PIPELINE:
  dvc repro                       # Runs only stages whose deps changed
  git add dvc.lock                # Commit the lock file (captures exact versions)
  git commit -m "data: update pipeline run v2"

COMPARE EXPERIMENT METRICS:
  dvc metrics show                # Show current metrics
  dvc metrics diff                # Compare metrics: current vs last commit
  dvc metrics diff v1.0.0         # Compare current vs a tag

EOF

# LESSON 06 — ML EXPERIMENT BRANCHING STRATEGY
# ==============================================================================

echo ""
echo "=== LESSON 06: Git Branching for ML Experiments ==="

cat << 'EOF'

PROBLEM: Data Scientists run MANY experiments (different models, features, hyperparams).
         Which ones are worth keeping? Which branch is the "best model"?

RECOMMENDED STRATEGY (combines GitHub Flow + DVC):

BRANCHES:
  main             → only PROVEN, production-ready code + models
  develop          → integration: all finished experiments merged here
  exp/lr-0.001     → experiment: try learning rate 0.001
  exp/xgboost      → experiment: try XGBoost instead of RF
  exp/add-features → experiment: add 3 new features

WORKFLOW PER EXPERIMENT:
  git checkout -b exp/dropout-0.3 develop
  # ... modify model.py, train, evaluate ...
  dvc push                          # Push data/model artifacts

  # Document results in commit message:
  git commit -m "exp: dropout=0.3 → F1=0.82 (prev best: 0.79)"
  # ↑ Include metric in commit message — searchable later!

  git push origin exp/dropout-0.3

  # If experiment WINS → PR to develop, then develop to main with tag
  # If experiment LOSES → branch stays for reference, you learned from it

FINDING YOUR BEST EXPERIMENT:
  git log --oneline --grep="exp:"  # Find all experiment commits
  git log --oneline | grep "F1"    # Find commits mentioning F1 score
  dvc metrics diff exp/dropout-0.3 exp/lr-0.001  # Compare two experiments!

NAMING CONVENTIONS FOR ML BRANCHES:
  exp/<description>        → experiments (will be deleted if not promoted)
  model/<version>          → model updates (e.g., model/xgboost-v2)
  data/<description>       → data pipeline changes
  feature/<name>           → code features
  fix/<description>        → bug fixes

EOF

cat << 'EOF'

Standard Conventional Commits + ML extensions:

PREFIXES:
  feat:     code: new feature (API endpoint, preprocessing step)
  fix:      code: bug fix
  data:     data pipeline change (new dataset, cleaning step)
  model:    model architecture or training change
  exp:      experiment result (include metrics in message!)
  eval:     evaluation or metrics update
  refactor: code cleanup (no behavior change)
  test:     adding or fixing tests
  ci:       GitHub Actions / CI change
  docs:     documentation
  chore:    dependencies, config, tooling

ML-SPECIFIC EXAMPLES:
  "feat: add StandardScaler to numeric preprocessing pipeline"
  "fix: handle NaN values in age column during imputation"
  "data: add 50K new customer records from Q1 2026 export"
  "model: switch from RandomForest to GradientBoosting"
  "exp: lr=0.001 batch=64 → F1=0.847 AUC=0.921 (best so far)"
  "eval: v2 model outperforms v1 on holdout — AUC +0.04"
  "ci: add model performance check to GitHub Actions pipeline"
  "chore: update scikit-learn to 1.4.0 for pipeline compatibility"

WHY INCLUDE METRICS IN COMMIT MESSAGES?
  → git log --oneline | grep "F1"   finds your best experiments instantly
  → teammates see results without running code
  → proves progress over time — great for portfolio/performance reviews
  → DVC can also store metrics files — but commit messages = human-readable log

GOOD COMMIT MESSAGE ANATOMY:
  exp: add attention mechanism to churn model

  - Architecture: BiLSTM + self-attention (512 units)
  - Training: 50 epochs, lr=0.001, batch=128 (GPU)
  - Results:
      AUC-ROC:  0.931 (prev: 0.892, +4.4%)
      F1-score: 0.864 (prev: 0.823, +5.0%)
      Precision: 0.871, Recall: 0.858
  - Data: 450K customers, 80/20 split, stratified
  - Next: try adding engineered features (see exp/feature-eng branch)

  Closes #42

EOF

TASK 1 — GITIGNORE FOR ML:
    Create a new project folder and add the ML .gitignore from LESSON 02.
    a) Create: data/train.csv (add some dummy data)
    b) git status → is train.csv ignored?
    c) Create: .env with API_KEY=secret123
    d) git status → is .env ignored?
    e) Create: model.pkl (empty file)
    f) git status → is model.pkl ignored?
    ALL THREE should be ignored. Fix .gitignore if any appear.

TASK 2 — PRE-COMMIT HOOKS:
    In any Python project:
    a) pip install pre-commit
    b) Create .pre-commit-config.yaml with at least:
         - trailing-whitespace hook
         - black hook
         - nbstripout hook (if you have notebooks)
    c) pre-commit install
    d) Create a Python file with bad formatting (inconsistent indentation, trailing spaces)
    e) git add + git commit → hooks should BLOCK and fix the issues automatically
    f) git add + git commit again → should succeed now

TASK 3 — GITHUB ACTIONS:
    a) Create a GitHub repo with a simple Python project.
    b) Add tests/test_model.py with 2 simple pytest tests.
    c) Create .github/workflows/ci.yml from LESSON 04.
    d) Push to GitHub.
    e) Go to your repo → Actions tab → watch the CI run.
    f) Intentionally break a test, push again → CI should FAIL (red).
    g) Fix the test, push → CI should PASS (green).

TASK 4 — DVC BASICS:
    a) pip install dvc
    b) In a Git repo: dvc init → git add .dvc → git commit "chore: init DVC"
    c) Add a CSV file: dvc add data.csv
    d) git add data.csv.dvc data/.gitignore
    e) git commit -m "data: add initial dataset"
    f) Modify the CSV (add a row), dvc add it again, git commit with metrics in message
    g) dvc diff → see what data changed between commits

TASK 5 — EXPERIMENT TRACKING:
    a) Create branch: exp/random-forest
    b) Train a RandomForest on any dataset, save F1 to metrics.json
    c) Commit: "exp: random-forest → F1=0.XXX"
    b) Create branch: exp/gradient-boosting
    d) Train a GradientBoosting, save F1 to metrics.json
    e) Commit: "exp: gradient-boosting → F1=0.YYY"
    f) Compare:
         git log --oneline | grep "exp:"     → see both experiments
         git diff exp/random-forest..exp/gradient-boosting -- metrics.json
    g) Which experiment won? PR the winner to develop.
