#        - A training job that takes 8 hours will be KILLED if you close your laptop.
#        - tmux solves this — your session lives on the server, not your connection.
#        - SSH tunneling lets you run Jupyter on a powerful remote GPU and access

# ==============================================================================
# LESSON 01 — SSH: SECURE SHELL (Connecting to Remote Servers)
# ==============================================================================
#
# TEACHER SAYS:
# SSH = Secure Shell. It lets you open a terminal session on a REMOTE machine.
# You type commands on your laptop — they run on a server in a data center.
#
# ANATOMY OF THE SSH COMMAND:
#   ssh  username@hostname_or_IP
#        │         └── IP address (e.g. 18.234.56.78) OR domain (ml-server.company.com)
#        └── your username ON the remote machine (not your laptop username!)
#
# DEFAULT: SSH uses port 22. Most firewalls allow this.
# FIRST CONNECT: SSH will ask "Are you sure you want to connect?" → type 'yes'
#   This saves the server's fingerprint to ~/.ssh/known_hosts to verify future connections.

# Connect to a remote server:
ssh student@192.168.1.100                # Connect as 'student' user to that IP
ssh ubuntu@18.234.56.78                  # Connect to an AWS EC2 Ubuntu instance
ssh alice@ml-server.company.com          # Connect using a hostname

# Connect on a non-standard port (some servers use 2222 for security):
ssh -p 2222 alice@ml-server.company.com

# Run a SINGLE command without staying connected:
ssh ubuntu@18.234.56.78 'ls ~/projects'   # List files and disconnect
ssh ubuntu@18.234.56.78 'df -h'           # Check disk space remotely
ssh ubuntu@18.234.56.78 'tail -50 training.log'  # Check training log from laptop

# Exit SSH session (get back to your laptop):
exit
# OR: Ctrl+D


# ==============================================================================
# LESSON 02 — SSH KEYS (Log In Without Typing a Password)
# ==============================================================================
#
# TEACHER SAYS:
# Typing a password every time is inconvenient and less secure.
# SSH KEYS are the professional way to authenticate.
#
# HOW SSH KEYS WORK:
#   1. You generate a KEY PAIR on your laptop:
#      - PRIVATE KEY (~/.ssh/id_rsa)  ← stays on your laptop ONLY. Never share!
#      - PUBLIC KEY (~/.ssh/id_rsa.pub) ← give this to servers you want to connect to.
#   2. The server holds your PUBLIC KEY in ~/.ssh/authorized_keys
#   3. When you SSH, the server challenges you with a puzzle only your PRIVATE KEY can solve
#   4. No password needed — math proves you own the private key.
#
# ANALOGY: Public key = padlock. Private key = the key that opens it.
#          You give padlocks to everyone. You keep the key to yourself.

# Step 1: Generate your SSH key pair (do this ONCE on your laptop):
ssh-keygen -t ed25519 -C "your_email@example.com"
# -t ed25519 → type: Ed25519 is modern and fast (preferred over RSA in 2024)
# -C → comment/label (helps you identify the key later)
# It will ask where to save (press Enter for default: ~/.ssh/id_ed25519)
# It will ask for a passphrase (optional but recommended for extra security)

# See your keys:
ls ~/.ssh/
# id_ed25519      ← private key (NEVER share this!)
# id_ed25519.pub  ← public key (safe to share)
cat ~/.ssh/id_ed25519.pub  # See your public key (copy this to servers!)

# Step 2: Copy your public key to the server (the easy way):
ssh-copy-id ubuntu@18.234.56.78
# This automatically adds your public key to ~/.ssh/authorized_keys on the server

# Manual alternative (if ssh-copy-id isn't available):
cat ~/.ssh/id_ed25519.pub | ssh ubuntu@18.234.56.78 'mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys'

# Step 3: Test — log in without password:
ssh ubuntu@18.234.56.78    # Should connect without asking for a password!


# ==============================================================================
# LESSON 03 — SSH CONFIG FILE (Never Remember Long Commands Again)
# ==============================================================================
#
# TEACHER SAYS:
# ssh ubuntu@18.234.56.78 -p 2222 -i ~/.ssh/my_aws_key.pem
# is painful to type every time. The SSH config file lets you define shortcuts.
# After setup: just type 'ssh myserver' and it knows everything automatically.

# Create/edit your SSH config file:
# On your LAPTOP, run:
mkdir -p ~/.ssh
chmod 700 ~/.ssh
# nano or vim ~/.ssh/config  ← uncomment this to edit in your terminal

# Example ~/.ssh/config file content:
cat << 'EOF'
# ~/.ssh/config — SSH connection shortcuts

# AWS EC2 ML Training server
Host mlserver
    HostName 18.234.56.78
    User ubuntu
    IdentityFile ~/.ssh/aws_ml_key.pem
    Port 22

# Company internal server (non-standard port)
Host work-server
    HostName ml-server.company.com
    User alice
    IdentityFile ~/.ssh/id_ed25519
    Port 2222

# Shortcut with forwarding (for Jupyter — explained in Lesson 05)
Host jupyter-server
    HostName 18.234.56.78
    User ubuntu
    IdentityFile ~/.ssh/aws_ml_key.pem
    LocalForward 8888 localhost:8888
EOF

# After saving the config file:
chmod 600 ~/.ssh/config    # Must be readable ONLY by you — SSH will refuse otherwise!

# Now connect with just the alias:
# ssh mlserver            ← equivalent to the long command above
# ssh work-server
# ssh jupyter-server      ← connects AND sets up port forwarding automatically


# ==============================================================================
# LESSON 04 — SCP & RSYNC (Transferring Files to/from Servers)
# ==============================================================================
#
# TEACHER SAYS:
# Three scenarios you'll face constantly:
#   1. Upload training data to server before a run
#   2. Download model outputs / trained weights back to your laptop
#   3. Sync your local code to the server (faster than git clone every time)
#
# SCP (Secure Copy) = simple file transfer via SSH
# rsync = smarter sync — only transfers CHANGED files (huge speed advantage!)

# SCP — syntax: scp [source] [destination]
# Use  user@host:path  for remote paths, just path for local

# Upload FROM laptop TO server:
scp mydata.csv ubuntu@18.234.56.78:~/data/
scp -r ./models/ ubuntu@18.234.56.78:~/project/models/   # -r for directories

# Download FROM server TO laptop:
scp ubuntu@18.234.56.78:~/models/churn_v2.pkl ./models/
scp -r ubuntu@18.234.56.78:~/results/ ./results/

# Using SSH config alias (much shorter!):
scp mydata.csv mlserver:~/data/
scp -r mlserver:~/results/ ./results/

# RSYNC — smarter sync (only copies changes):
# Syntax: rsync [options] [source] [destination]

# Sync your local project TO server (only changed files):
rsync -avz ./my_project/ mlserver:~/my_project/
# -a = archive mode (preserves permissions, timestamps, recursive)
# -v = verbose (shows what's being transferred)
# -z = compress during transfer (faster over slow connections)

# Sync server results BACK to laptop:
rsync -avz mlserver:~/results/ ./results/

# rsync with EXCLUSIONS (don't sync data files or caches):
rsync -avz --exclude='data/' --exclude='__pycache__/' --exclude='*.pyc' \
    ./my_project/ mlserver:~/my_project/

# Dry run first (see what WOULD be synced, without actually syncing):
rsync -avz --dry-run ./my_project/ mlserver:~/my_project/

# Delete remote files that were deleted locally (true sync):
rsync -avz --delete ./my_project/ mlserver:~/my_project/


# ==============================================================================
# LESSON 05 — SSH TUNNELING (Run Jupyter on a Server, Use It Locally)
# ==============================================================================
#
# TEACHER SAYS:
# Scenario: You have a powerful GPU server (8xA100s) in the cloud.
# You want to run Jupyter Notebook on THAT server (to use the GPUs)
# but access it in YOUR browser on your laptop.
#
# SSH TUNNELING = "port forwarding"
# It creates a secure pipe: your laptop's port 8888 → server's port 8888
# Your browser thinks Jupyter is local, but it's actually running remotely!
#
# STEP 1: Start Jupyter on the server (no browser since server has no display)
# (Run this on the server after SSH-ing in)
#   jupyter notebook --no-browser --port=8888
#
# STEP 2: On your LAPTOP, create the tunnel:
ssh -L 8888:localhost:8888 ubuntu@18.234.56.78
# -L = "Local forwarding"
# 8888:localhost:8888 = "my local port 8888 → server's localhost:8888"
#
# STEP 3: Open your laptop browser → http://localhost:8888
# You'll see Jupyter — but it's running on the server using its GPUs!

# One-liner: start tunnel in background (-N = don't execute command, just forward):
ssh -N -L 8888:localhost:8888 mlserver &
# Now open browser → localhost:8888

# Multiple port forwards in one command:
ssh -L 8888:localhost:8888 -L 6006:localhost:6006 mlserver
# Also forwards TensorBoard port 6006!

# Check your own open tunnels:
ps aux | grep ssh


# ==============================================================================
# LESSON 06 — TMUX (Never Lose a Running Process When Connection Drops)
# ==============================================================================
#
# TEACHER SAYS:
# WITHOUT tmux: You SSH into a server, start training (8 hours), your laptop
#   sleeps or disconnects → the SSH session dies → training STOPS.
#   8 hours of GPU time, wasted.
#
# WITH tmux: Your "session" lives on the SERVER, not in your SSH connection.
#   You can disconnect and reconnect. The session (and your training) keeps running.
#   tmux also lets you have multiple "windows" (like browser tabs) in one terminal,
#   and split the screen into "panes" (like VS Code panels).
#
# MENTAL MODEL:
#   tmux SERVER → lives on the remote machine
#   tmux SESSION → one workspace (can have many windows)
#   tmux WINDOW  → one "tab" (like a browser tab)
#   tmux PANE    → one split section of a window
#
# HOW IT WORKS:
#   You type commands inside a tmux window.
#   You "detach" from tmux (Ctrl+B, D) → tmux session keeps running on the server.
#   You disconnect SSH → nothing stops.
#   You reconnect later → reattach to the session → everything is there.
#
# All tmux commands start with PREFIX = Ctrl+B
#   (Press Ctrl+B, release, THEN press the next key)

# Install tmux:
sudo apt install tmux            # Ubuntu/Debian
# brew install tmux              # macOS

# ==============================================================================
# TMUX SESSION MANAGEMENT
# ==============================================================================

# Start a new named session (ALWAYS name your sessions!):
tmux new -s training             # Start session called "training"
tmux new -s jupyter              # Start session called "jupyter"

# Inside tmux: start your ML training job:
#   python train.py --epochs=100 --lr=0.001 > train.log 2>&1

# Detach from session (training keeps running!):
# Press: Ctrl+B, then D
# You're back to your normal shell. The tmux session is still alive.

# List all running sessions:
tmux ls
# Output:
# training: 1 windows (created Mon Apr 21 10:30:00 2026)
# jupyter:  1 windows (created Mon Apr 21 09:00:00 2026)

# Re-attach to a session (after reconnecting SSH):
tmux attach -t training          # Rejoin "training" session
tmux attach -t jupyter           # Rejoin "jupyter" session
tmux a                           # Attach to most recently used session

# Rename a session (if you forgot to name it):
tmux rename-session -t 0 training

# Kill a session (when you're done with it):
tmux kill-session -t training

# ==============================================================================
# TMUX WINDOWS (Tabs)
# ==============================================================================
# All window commands start with Ctrl+B

# Create a new window:                  Ctrl+B, C
# Switch to next window:                Ctrl+B, N
# Switch to previous window:            Ctrl+B, P
# Switch to window by number:           Ctrl+B, 0  (or 1, 2, 3...)
# Rename current window:                Ctrl+B, ,
# List windows:                         Ctrl+B, W  (then arrow keys to select)
# Close current window:                 Ctrl+B, &  (confirms before closing)

# ==============================================================================
# TMUX PANES (Split Screen)
# ==============================================================================

# Split window HORIZONTALLY (top/bottom):  Ctrl+B, "   (that's the " key)
# Split window VERTICALLY (left/right):    Ctrl+B, %
# Switch between panes:                    Ctrl+B, Arrow keys
# Resize pane:                             Ctrl+B, Hold Arrow key
# Close current pane:                      Ctrl+B, X  (then confirm with 'y')
# Zoom in on current pane (full screen):   Ctrl+B, Z  (press again to zoom out)
# ==============================================================================

# TYPICAL ML TRAINING WORKFLOW WITH TMUX:
# 1. SSH into server
# 2. tmux new -s ml_run
# 3. Split into 3 panes:
#    Pane 1 (top-left):  training script output
#    Pane 2 (top-right): watch -n 5 nvidia-smi   (live GPU monitor)
#    Pane 3 (bottom):    tail -f train.log        (log file)
# 4. Start training in pane 1
# 5. Ctrl+B, D → detach (training runs on server!)
# 6. Close laptop, go home
# 7. Come back tomorrow: ssh mlserver → tmux attach -t ml_run
# 8. See your training results as if you never left!

# Quick one-line tmux setup for ML monitoring:
cat << 'SCRIPT'
# Run this on the server to set up your ML monitoring layout:
tmux new -s training \; \
  send-keys 'python train.py > train.log 2>&1' Enter \; \
  split-window -v \; \
  send-keys 'watch -n 2 nvidia-smi' Enter \; \
  split-window -h \; \
  send-keys 'tail -f train.log' Enter \; \
  select-pane -t 0
SCRIPT


# ==============================================================================
# LESSON 07 — DISK & NETWORK MONITORING (ML Engineer Essentials)
# ==============================================================================
#
# TEACHER SAYS:
# Three things kill ML server jobs without warning:
#   1. Disk full → training crashes when it can't write checkpoints
#   2. GPU OOM (out of memory) → training crashes with CUDA error
#   3. Network timeout → data loading stalls
# Learn to spot these BEFORE they become problems.

# Check available disk space:
df -h                            # All mounted filesystems, human-readable
df -h /                          # Just the root filesystem
df -h /data                      # Check the data partition specifically

# Find what's eating your disk:
du -sh *                         # Size of each item in current directory
du -sh --max-depth=1 ~/          # Top-level disk usage of home directory
du -sh ~/.cache                  # How big is your cache?
du -sh ~/data/* | sort -h        # Sort by size (smallest first)
du -sh ~/data/* | sort -rh       # Sort by size (LARGEST first) ← most useful!

# Clean up disk space:
find . -name "*.pyc" -delete               # Delete compiled Python files
find . -type d -name "__pycache__" -exec rm -rf {} +   # Delete all pycache
rm -rf ~/.cache/pip                        # Clear pip download cache

# Monitor network traffic:
# Check if data is downloading (is wget still running?):
ps aux | grep wget
# See network connections:
ss -tlnp                                   # What ports are listening?
# Or older tool:
netstat -tlnp 2>/dev/null                  # Same thing (if installed)

# Which process is using a port?
ss -tlnp | grep 8888                       # Who's on port 8888? (Jupyter?)
ss -tlnp | grep 8000                       # Who's on port 8000? (FastAPI?)

# Check if your API server is actually running:
curl -s http://localhost:8000/health && echo "✓ Server is up" || echo "✗ Server is down"


# ==============================================================================
# LESSON 08 — DISK & ARCHIVE TOOLS (tar, gzip, zip)
# ==============================================================================
#
# TEACHER SAYS:
# Datasets come compressed. Model checkpoints need to be archived.
# These commands handle all compressed file formats.
#
# tar  = creates/extracts archives (like a box that holds files)
# gzip = compresses individual files (.gz extension)
# zip  = ZIP format (Windows-compatible)
#
# MOST COMMON PATTERN: .tar.gz (a "tarball") = tar archive + gzip compression
# OPTIONS for tar: c=create, x=extract, z=gzip, v=verbose, f=filename
#   MEMORIZE:  c=compress, x=extract, z=zip, v=verbose, f=file
#              CVF → Create Very Fine archive 😄

# Create a .tar.gz archive:
tar -czf models_backup.tar.gz ./models/          # Archive the models/ folder
tar -czf data_v1.tar.gz data/raw/ data/processed/ # Archive multiple folders

# Extract a .tar.gz archive:
tar -xzf models_backup.tar.gz                    # Extract in current directory
tar -xzf models_backup.tar.gz -C /target/path/  # Extract to specific location

# List contents without extracting:
tar -tzf models_backup.tar.gz                    # Preview what's inside

# Compress/decompress single files:
gzip large_file.csv                              # Creates large_file.csv.gz, removes original
gzip -d large_file.csv.gz                        # Decompress (or: gunzip large_file.csv.gz)
gzip -k large_file.csv                           # Keep original after compressing (-k flag)

# ZIP format (for sharing with Windows users):
zip -r archive.zip ./my_folder/                  # Create zip
unzip archive.zip                                # Extract zip
unzip -l archive.zip                             # List contents

# Check file type (what kind of compressed file is this?):
file unknown_archive                             # Shows: "POSIX tar archive (gzip compressed)"

# Pro tip — compress and transfer in one pipeline:
tar -czf - ./models/ | ssh mlserver 'tar -xzf - -C ~/backup/'
# Creates archive → pipes directly into SSH → extracts on server
# Never writes a local file! Great when disk space is tight.


# ==============================================================================
# LESSON 09 — jq: JSON PROCESSING FROM THE TERMINAL
# ==============================================================================
#
# TEACHER SAYS:
# APIs return JSON. Your ML pipeline configs are JSON. Your metrics are JSON.
# jq is to JSON what awk is to CSV — it lets you filter, extract, and transform
# JSON directly in the terminal without writing Python.
#
# WHY LEARN THIS:
#   When debugging at 2 AM on a production server, you don't want to
#   write a Python script just to read one value from a JSON file.
#   jq lets you do it in one line.

# Install jq:
sudo apt install jq              # Ubuntu/Debian
# brew install jq                # macOS

# Create a sample JSON file to practice with:
cat > metrics.json << 'EOF'
{
  "experiment": "churn_v3",
  "model": "random_forest",
  "hyperparams": {
    "n_estimators": 200,
    "max_depth": 10,
    "learning_rate": 0.001
  },
  "results": {
    "train_f1": 0.923,
    "val_f1": 0.847,
    "test_f1": 0.841,
    "auc_roc": 0.912
  },
  "runs": [
    {"epoch": 1, "loss": 0.45, "f1": 0.72},
    {"epoch": 2, "loss": 0.31, "f1": 0.81},
    {"epoch": 3, "loss": 0.22, "f1": 0.85}
  ]
}
EOF

# . means "the whole document" (the root):
cat metrics.json | jq '.'                        # Pretty-print the whole file

# Access a field:
cat metrics.json | jq '.model'                   # "random_forest"
cat metrics.json | jq '.experiment'              # "churn_v3"

# Nested fields:
cat metrics.json | jq '.results.val_f1'          # 0.847
cat metrics.json | jq '.hyperparams.n_estimators' # 200

# Multiple fields (output as object):
cat metrics.json | jq '{model: .model, val_f1: .results.val_f1}'

# Array access:
cat metrics.json | jq '.runs[0]'                 # First run object
cat metrics.json | jq '.runs[-1]'                # Last run object
cat metrics.json | jq '.runs[].loss'             # All loss values

# Filter array elements:
cat metrics.json | jq '.runs[] | select(.f1 > 0.80)'  # Runs where f1 > 0.80

# Raw output (no quotes — useful for piping to other commands):
cat metrics.json | jq -r '.model'                # random_forest (no quotes)

# Read from file directly (no cat needed):
jq '.results' metrics.json

# Real-world use: extract val_f1 from many experiment files and compare:
for f in experiments/*.json; do
    echo "$f: $(jq -r '.results.val_f1' "$f")"
done
# Output:
# experiments/run1.json: 0.832
# experiments/run2.json: 0.847
# experiments/run3.json: 0.851

# LESSON 10 — THE COMPLETE ML WORKFLOW ON A REMOTE SERVER
# ==============================================================================
#
# TEACHER SAYS:
# This is the full professional workflow. Every step matters.
# Do this once and you'll never need a tutorial again.

cat << 'WORKFLOW'
=================================================================
 COMPLETE REMOTE ML SERVER WORKFLOW
=================================================================

STEP 1: SET UP SSH KEY (one-time setup)
  Local:
    ssh-keygen -t ed25519 -C "your_email@example.com"
    ssh-copy-id ubuntu@YOUR_SERVER_IP

STEP 2: ADD SERVER TO SSH CONFIG (one-time setup)
  Edit ~/.ssh/config:
    Host mlserver
        HostName YOUR_SERVER_IP
        User ubuntu
        IdentityFile ~/.ssh/id_ed25519

STEP 3: SYNC YOUR CODE TO SERVER
  Local:
    rsync -avz --exclude='data/' --exclude='__pycache__/' \
      ./my_project/ mlserver:~/my_project/

STEP 4: SSH IN AND START TMUX
  Local:
    ssh mlserver
  On server:
    tmux new -s training

STEP 5: SET UP ENVIRONMENT ON SERVER (first time)
  On server (inside tmux):
    cd ~/my_project
    python3 -m venv venv
    source venv/bin/activate
    pip install -r requirements.txt

STEP 6: SET UP MONITORING PANES
  In tmux:
    Ctrl+B, "         → split horizontally
    In bottom pane:
    watch -n 2 nvidia-smi    → live GPU monitor

    Ctrl+B, Up arrow         → go back to top pane
    Ctrl+B, %                → split vertically
    In right pane:
    tail -f training.log     → live log

STEP 7: START TRAINING (in the main pane)
    python train.py > training.log 2>&1

STEP 8: DETACH AND WALK AWAY
    Ctrl+B, D                → detach from tmux
    exit                     → disconnect SSH
    → Training continues on server!

STEP 9: CHECK PROGRESS LATER (from anywhere)
  Local:
    ssh mlserver 'tail -20 ~/my_project/training.log'
    OR:
    ssh mlserver
    tmux attach -t training

STEP 10: DOWNLOAD RESULTS
  Local:
    rsync -avz mlserver:~/my_project/models/ ./models/
    rsync -avz mlserver:~/my_project/results/ ./results/

STEP 11: CHECK METRICS
  Local (if metrics saved as JSON):
    jq '.results' models/churn_v3_metrics.json
=================================================================
WORKFLOW
