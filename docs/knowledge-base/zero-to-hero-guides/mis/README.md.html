Zero-to-Hero Guide: Management Information Systems (MIS) in Python

The enterprise layer. Supply Chain is the physical movement of goods. MIS is the digital nervous system that tracks, manages, and reports on the entire company. As an AI/Data Engineer with an MIS background, you don't just build models — you build the internal tools and dashboards that executives use to run the business.
This is where your Engineering Management degree earns a salary premium over a pure CS graduate.

The Three Technical Pillars

Pillar 1: Data Architecture & ETL  → centralise all company data in one place
Pillar 2: Business Intelligence     → turn that data into executive dashboards
Pillar 3: Enterprise Automation     → make the system act on its own findings

The capstone result: a Custom MIS Platform — not a "project," a platform — with a login screen, live KPI dashboard, ML-powered alerts, and automated notifications. That is what gets you hired as an Enterprise AI Architect.

Pillar 1 — Data Architecture & ETL (The Foundation)

The Business Problem: The marketing team's Excel says revenue is $1.2M. The factory's ERP says $980k. Finance says $1.1M. Nobody trusts anyone. Your job: build the single source of truth.

The Data Science Skill: ETL (Extract, Transform, Load) + Data Warehouse design

1a. Star Schema — The Industry Standard Database Design

Before writing a single line of ETL code, design the database correctly. The Kimball Methodology uses Star Schemas: one central fact table surrounded by dimension tables.

                    dim_product
                        │
dim_date ──── fact_sales ──── dim_store
                        │
                    dim_customer
  • Fact table (fact_sales): stores measurable events — each row is one sale, one sensor reading, one delivery
  • Dimension tables (dim_product, dim_store, etc.): store descriptive attributes — product name, category, store region

Why does this matter? Managers run queries like:

"Show me total revenue by product category, by month, for the last 2 years."

A star schema makes this query a single SQL JOIN. A poorly designed database makes it impossible without an engineer rewriting it.

-- Example: sales fact table + dimensions
CREATE TABLE dim_date (
    date_key    INTEGER PRIMARY KEY,  -- YYYYMMDD integer key (faster than DATE for joins)
    full_date   DATE,
    year        INTEGER,
    quarter     INTEGER,
    month       INTEGER,
    week        INTEGER,
    day_of_week VARCHAR(10)
);

CREATE TABLE dim_product (
    product_key     SERIAL PRIMARY KEY,
    product_id      VARCHAR(50) UNIQUE,
    product_name    VARCHAR(200),
    category        VARCHAR(100),
    subcategory     VARCHAR(100),
    unit_cost_usd   DECIMAL(10,2)
);

CREATE TABLE dim_store (
    store_key   SERIAL PRIMARY KEY,
    store_id    VARCHAR(50) UNIQUE,
    store_name  VARCHAR(200),
    city        VARCHAR(100),
    region      VARCHAR(100),
    country     VARCHAR(100)
);

CREATE TABLE fact_sales (
    sale_id         SERIAL PRIMARY KEY,
    date_key        INTEGER REFERENCES dim_date(date_key),
    product_key     INTEGER REFERENCES dim_product(product_key),
    store_key       INTEGER REFERENCES dim_store(store_key),
    units_sold      INTEGER,
    revenue_usd     DECIMAL(12,2),
    cost_usd        DECIMAL(12,2),
    profit_usd      DECIMAL(12,2)  -- computed column: revenue - cost
);

-- Manager query: revenue by category per month
SELECT
    d.year,
    d.month,
    p.category,
    SUM(f.revenue_usd)  AS total_revenue,
    SUM(f.profit_usd)   AS total_profit,
    SUM(f.units_sold)   AS units_sold
FROM fact_sales f
JOIN dim_date    d ON f.date_key    = d.date_key
JOIN dim_product p ON f.product_key = p.product_key
GROUP BY d.year, d.month, p.category
ORDER BY d.year, d.month, p.category;

1b. ETL Pipeline in Python

An ETL pipeline extracts data from source systems (APIs, CSVs, other databases), transforms it (cleans, reshapes, joins), and loads it into your data warehouse.

import pandas as pd
import sqlalchemy
from sqlalchemy import create_engine
from datetime import datetime
import logging

logger = logging.getLogger(__name__)


class ETLPipeline:
    """
    A reusable ETL base class.

    Design principle: each pipeline run should be IDEMPOTENT — running it
    twice should produce the same result as running it once. This means:
    - Use UPSERT (INSERT ... ON CONFLICT UPDATE) instead of plain INSERT
    - Track the last successful run timestamp and only pull new records
    - Never delete rows from the warehouse; add an 'is_active' flag instead

    This matters in production because pipelines fail mid-run. You need
    to be able to restart them safely.
    """

    def __init__(self, warehouse_url: str):
        """
        Args:
            warehouse_url: SQLAlchemy connection string, e.g.
                'postgresql://user:password@localhost:5432/warehouse'
        """
        self.engine = create_engine(warehouse_url)
        self.run_log: list[dict] = []

    def extract(self, source) -> pd.DataFrame:
        """
        Pull raw data from the source system.

        Sources you will encounter:
        - CSV / Excel files:    pd.read_csv() or pd.read_excel()
        - REST API:             requests.get(url, headers=auth).json()
        - Another database:     pd.read_sql(query, other_engine)
        - IoT/MQTT stream:      see phase-4-iot-hardware/
        - S3 / cloud storage:   pd.read_csv("s3://bucket/file.csv")

        TODO: implement for your specific source
        """
        raise NotImplementedError

    def transform(self, raw_df: pd.DataFrame) -> pd.DataFrame:
        """
        Clean and reshape the raw data to match the warehouse schema.

        Common transformations:
        - Standardise column names (snake_case, no spaces)
        - Parse date strings into datetime objects
        - Handle nulls: fill, drop, or flag with a sentinel value
        - Derive new columns (profit = revenue - cost)
        - Map source IDs to warehouse surrogate keys
        - Validate data ranges (price > 0, quantity >= 0)

        TODO: implement for your specific schema
        """
        raise NotImplementedError

    def load(self, clean_df: pd.DataFrame, table_name: str, if_exists: str = "append"):
        """
        Writes the transformed data to the warehouse.

        Args:
            if_exists: 'append' for incremental loads, 'replace' for full refresh

        TODO: clean_df.to_sql(table_name, self.engine, if_exists=if_exists, index=False)

        For UPSERT (recommended in production):
            Use raw SQL with ON CONFLICT DO UPDATE, or
            use the 'upsert' pattern: load to a staging table, then MERGE/UPSERT
            from staging into the final table.
        """
        clean_df.to_sql(table_name, self.engine, if_exists=if_exists, index=False)
        logger.info(f"Loaded {len(clean_df)} rows into {table_name}")

    def run(self, source, table_name: str):
        """Full ETL run with timing and error logging."""
        start = datetime.utcnow()
        try:
            raw  = self.extract(source)
            clean = self.transform(raw)
            self.load(clean, table_name)
            status = "SUCCESS"
            rows = len(clean)
        except Exception as e:
            logger.error(f"ETL failed: {e}")
            status = "FAILED"
            rows = 0
            raise
        finally:
            self.run_log.append({
                "table": table_name,
                "started_at": start.isoformat(),
                "duration_s": (datetime.utcnow() - start).total_seconds(),
                "rows_loaded": rows,
                "status": status,
            })


class SalesCSVPipeline(ETLPipeline):
    """
    Concrete ETL: reads a sales CSV and loads it into the fact_sales table.
    Demonstrates how to subclass ETLPipeline for a specific source.
    """

    def extract(self, csv_path: str) -> pd.DataFrame:
        df = pd.read_csv(csv_path)
        logger.info(f"Extracted {len(df)} rows from {csv_path}")
        return df

    def transform(self, raw_df: pd.DataFrame) -> pd.DataFrame:
        df = raw_df.copy()

        # Standardise column names
        df.columns = [c.lower().replace(" ", "_") for c in df.columns]

        # Parse dates
        df["sale_date"] = pd.to_datetime(df["sale_date"])
        df["date_key"]  = df["sale_date"].dt.strftime("%Y%m%d").astype(int)

        # Derive profit
        df["profit_usd"] = df["revenue_usd"] - df["cost_usd"]

        # Drop rows with null product or store IDs (cannot join without them)
        df = df.dropna(subset=["product_id", "store_id"])

        # Validate ranges
        df = df[df["units_sold"] >= 0]
        df = df[df["revenue_usd"] >= 0]

        return df[["date_key", "product_id", "store_id", "units_sold",
                   "revenue_usd", "cost_usd", "profit_usd"]]

1c. Automating ETL with Celery Beat

In production, your ETL pipeline runs on a schedule — not manually. You already learn Celery in Month 6. Here is how it connects to MIS:

# In your Django app's tasks.py:

from celery import shared_task
from celery.schedules import crontab
from .etl_pipelines import SalesCSVPipeline

@shared_task
def nightly_sales_etl():
    """
    Runs every night at 2 AM to load the previous day's sales into the warehouse.
    Triggered by Celery Beat — no human intervention required.
    """
    pipeline = SalesCSVPipeline(warehouse_url="postgresql://...")
    pipeline.run(
        source="/data/exports/sales_yesterday.csv",
        table_name="fact_sales",
    )
    return {"rows_loaded": pipeline.run_log[-1]["rows_loaded"]}

2b. Plotly Dash Embedded in Django

For the capstone dashboard that lives inside your Django app (not a separate Metabase server), use Plotly. It integrates with Django templates and produces interactive charts.

# views.py — the Django view that powers the executive dashboard

import plotly.graph_objects as go
import plotly.utils
import json
from django.shortcuts import render
from .models import SensorReading, TransferPlan

def executive_dashboard(request):
    """
    Renders the executive dashboard.
    Charts are built server-side with Plotly and passed as JSON to the template.
    The template renders them with Plotly.js — no page reload needed.
    """
    # --- 1. Temperature trend chart (last 72 hours) ---
    readings = SensorReading.objects.filter(
        sensor_id="COOLER-01"
    ).order_by("-timestamp")[:144]  # 72h * 2 readings/hour

    times  = [r.timestamp for r in reversed(list(readings))]
    temps  = [r.temperature_c for r in reversed(list(readings))]

    temp_fig = go.Figure()
    temp_fig.add_trace(go.Scatter(
        x=times, y=temps,
        mode="lines",
        line=dict(color="#2196F3", width=2),
        name="Temperature (°C)"
    ))
    temp_fig.add_hline(
        y=8.0, line_dash="dash", line_color="red",
        annotation_text="Critical threshold (8°C)"
    )
    temp_fig.update_layout(
        title="Cold-Chain Temperature — Last 72 Hours",
        xaxis_title="Time",
        yaxis_title="°C",
        template="plotly_white",
        height=350,
    )

    # --- 2. KPI cards ---
    latest = SensorReading.objects.filter(sensor_id="COOLER-01").latest("timestamp")
    failure_pct = round(latest.failure_prob_48h * 100, 1)
    financial_risk = round(latest.financial_risk_usd, 0)

    # Colour logic for the KPI card
    if failure_pct < 20:
        risk_colour = "success"   # Bootstrap green
    elif failure_pct < 40:
        risk_colour = "warning"   # Bootstrap yellow
    else:
        risk_colour = "danger"    # Bootstrap red

    # --- 3. Transfer plan (if active) ---
    latest_plan = TransferPlan.objects.order_by("-created_at").first()

    context = {
        "temp_chart_json": json.dumps(temp_fig, cls=plotly.utils.PlotlyJSONEncoder),
        "current_temp":    latest.temperature_c,
        "failure_pct":     failure_pct,
        "financial_risk":  f"${financial_risk:,.0f}",
        "risk_colour":     risk_colour,
        "alert_active":    failure_pct >= 40,
        "transfer_plan":   latest_plan,
    }
    return render(request, "dashboard/executive.html", context)

The Data Science Skill: Workflow automation + external API integration

3a. Automated Email Alerts via Django

# In your Celery task — after run_decision_pipeline() returns a high-risk result

from django.core.mail import send_mail
from django.conf import settings

def send_failure_alert(sensor_id: str, failure_prob: float, financial_risk: float, plan: dict):
    """
    Sends a high-priority email to the maintenance team when failure risk > 40%.

    Django's send_mail() works with any SMTP provider.
    For production: use SendGrid (free tier: 100 emails/day) or AWS SES.

    In settings.py:
        EMAIL_BACKEND = 'django.core.mail.backends.smtp.EmailBackend'
        EMAIL_HOST = 'smtp.sendgrid.net'
        EMAIL_PORT = 587
        EMAIL_USE_TLS = True
        EMAIL_HOST_USER = 'apikey'
        EMAIL_HOST_PASSWORD = os.environ['SENDGRID_API_KEY']  # never hard-code
    """
    subject = f"[HIGH PRIORITY] Cooling Failure Risk: {failure_prob*100:.0f}% — {sensor_id}"

    plan_summary = "\n".join(
        f"  • {a['item_id']} → {a['to_warehouse']} (cost: ${a['cost']:.0f})"
        for a in plan.get("assignments", [])
    ) if plan else "  No transfer plan generated."

    body = f"""
AUTOMATED ALERT — Cold-Chain Monitoring System

Sensor:            {sensor_id}
Failure Risk (48h): {failure_prob*100:.1f}%
Financial Risk:    ${financial_risk:,.0f}

Recommended Transfer Plan (Total Cost: ${plan.get('total_cost_usd', 0):,.0f}):
{plan_summary}

Log in to the dashboard for full details: https://your-app.com/dashboard/

This is an automated message. Do not reply.
"""

    send_mail(
        subject=subject,
        message=body,
        from_email=settings.DEFAULT_FROM_EMAIL,
        recipient_list=["maintenance-manager@company.com"],
        fail_silently=False,
    )


# 3b. Slack alert (alternative — often preferred in modern companies)

import requests
import os

def send_slack_alert(failure_prob: float, financial_risk: float):
    """
    Sends a Slack message to the #maintenance-alerts channel.

    Setup:
    1. Go to api.slack.com/apps → Create New App → Incoming Webhooks
    2. Activate incoming webhooks → Add to workspace → copy the webhook URL
    3. Store the URL in an environment variable (never hard-code it)
    """
    webhook_url = os.environ.get("SLACK_WEBHOOK_URL")
    if not webhook_url:
        return

    colour = "danger" if failure_prob > 0.4 else "warning"
    payload = {
        "attachments": [{
            "color": colour,
            "title": f":rotating_light: Cooling Failure Risk: {failure_prob*100:.0f}%",
            "text": (
                f"*Financial Risk:* ${financial_risk:,.0f}\n"
                f"*Action:* Transfer plan generated. "
                f"<https://your-app.com/dashboard/|View Dashboard>"
            ),
            "footer": "Cold-Chain MIS Platform",
        }]
    }
    requests.post(webhook_url, json=payload, timeout=5)

3b. Apache Airflow — Orchestrating Complex Workflows

When your ETL pipeline has 10 steps and some depend on others finishing first, Celery Beat becomes hard to manage. Airflow is the industry standard for orchestrating these DAGs (Directed Acyclic Graphs) of tasks.

# A simple Airflow DAG: nightly ETL → run ML model → send report

from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime, timedelta

default_args = {
    "owner": "data-team",
    "retries": 2,
    "retry_delay": timedelta(minutes=5),
    "email_on_failure": True,
    "email": ["data-team@company.com"],
}

with DAG(
    dag_id="nightly_mis_pipeline",
    default_args=default_args,
    description="ETL → ML scoring → executive report",
    schedule_interval="0 2 * * *",   # 2 AM every night (cron syntax)
    start_date=datetime(2024, 1, 1),
    catchup=False,
    tags=["mis", "production"],
) as dag:

    extract_sales = PythonOperator(
        task_id="extract_sales_data",
        python_callable=run_sales_etl,
    )

    score_inventory = PythonOperator(
        task_id="score_inventory_risk",
        python_callable=run_ml_scoring,
    )

    send_report = PythonOperator(
        task_id="send_executive_report",
        python_callable=send_daily_report_email,
    )

    # Dependency chain: ETL must finish before ML scoring, which must finish before report
    extract_sales >> score_inventory >> send_report

Interview Talking Points

QuestionWhat to say
"What is a Star Schema and why does it matter?""A star schema separates measurable facts (sales, sensor readings) from descriptive dimensions (product, store, date) into separate tables. It matters because it lets any analyst run queries like 'revenue by region by month' with simple JOINs — no data engineering degree needed. The alternative is a normalized schema that requires 10-table JOINs for every business question."
"What is ETL and what makes a good ETL pipeline?""ETL is Extract-Transform-Load: pull data from source systems, clean and reshape it, then load it into the warehouse. A good pipeline is idempotent (safe to re-run), observable (every run is logged), and incremental (only processes new data, not the full history every night). Celery Beat handles scheduling; SQLAlchemy handles the DB writes."
"How do you present ML results to non-technical executives?""I never show a confusion matrix to an executive. I translate it: a 92% recall on failure prediction means 'we catch 92% of failures before they happen, and each failure costs $15,000 in spoilage, so this model is worth $X per year in avoided losses.' I build a dashboard with one number — Financial Risk ($) — coloured green, yellow, or red. That's the entire ML system from a business perspective."
"What is the difference between Celery and Airflow?""Celery is a task queue — it runs individual jobs asynchronously, like a background ML scoring task triggered by a web request. Airflow is a workflow orchestrator — it manages pipelines of tasks where step 2 depends on step 1 finishing successfully, with retries, alerts, and a visual DAG monitor. For small ETL jobs, Celery Beat is enough. When you have 20+ interdependent pipeline steps, you need Airflow."
