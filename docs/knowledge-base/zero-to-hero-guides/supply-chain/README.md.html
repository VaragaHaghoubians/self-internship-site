Zero-to-Hero Guide: Supply Chain AI in Python

Your target industry. Your IE background means you already understand the physics of a supply chain — how factories, warehouses, and transport networks connect. This guide shows you how to express that knowledge in Python and ML so companies like Amazon, FedEx, and Walmart see you as a domain expert, not a generic data scientist.

The Three Technical Pillars

Pillar 1: Demand Forecasting    → predict WHAT to order and HOW MUCH
Pillar 2: Inventory Optimization → decide HOW MUCH to keep in stock
Pillar 3: Route Optimization     → figure out HOW to deliver it

All three are required for a real supply chain. Most data scientists only know one. Knowing all three and how they chain together is what gets you hired at Amazon Logistics.

Pillar 1 — Demand Forecasting (Predicting the Future)

The Business Problem: A company needs to know how many units to produce or order before customers ask for them. Over-forecast → wasted inventory. Under-forecast → stockout, lost revenue.

The Data Science Skill: Time-Series Forecasting

1a. Statistical Models: ARIMA and SARIMA

ARIMA (AutoRegressive Integrated Moving Average) is the classical method. You already covered this in knowledge-base/learning-journey/month-03-machine-learning/ — here it is applied to demand data.

import pandas as pd
from statsmodels.tsa.statespace.sarimax import SARIMAX
from sklearn.metrics import mean_absolute_error
import warnings
warnings.filterwarnings("ignore")

def fit_sarima_demand(train_series: pd.Series, order=(1,1,1), seasonal_order=(1,1,1,52)):
    """
    Fits a SARIMA model to weekly demand data.

    order = (p, d, q)   — AR, differencing, MA terms
    seasonal_order = (P, D, Q, m) — seasonal terms; m=52 for weekly data, m=12 for monthly

    When to use SARIMA vs Prophet:
    - SARIMA: better when you have <2 years of data and clean seasonality
    - Prophet: better when you have multiple seasonalities, holidays, or missing data

    TODO: fit the model and return (model_fit, aic_score)
    """
    model = SARIMAX(
        train_series,
        order=order,
        seasonal_order=seasonal_order,
        enforce_stationarity=False,
        enforce_invertibility=False,
    )
    model_fit = model.fit(disp=False)
    return model_fit, model_fit.aic


def forecast_demand(model_fit, steps: int) -> pd.Series:
    """
    Returns the next `steps` periods of demand forecast with confidence intervals.

    TODO: forecast = model_fit.get_forecast(steps=steps)
    TODO: return forecast.predicted_mean
    """
    forecast = model_fit.get_forecast(steps=steps)
    pred_mean = forecast.predicted_mean
    pred_ci   = forecast.conf_int()  # confidence interval
    return pred_mean, pred_ci

1b. Modern ML: Prophet (Meta's Forecasting Library)

Prophet handles multiple seasonalities, holidays, and missing data — ARIMA cannot. For retail demand forecasting, Prophet is almost always the first tool to try.

# pip install prophet
from prophet import Prophet
import pandas as pd

def fit_prophet_demand(df: pd.DataFrame, country_holidays: str = None):
    """
    Fits a Prophet model to demand data.

    Args:
        df: DataFrame with two columns: 'ds' (datetime) and 'y' (demand value)
        country_holidays: ISO country code (e.g. 'US', 'BR') to add public holidays

    Prophet automatically detects:
    - Weekly seasonality (higher demand on weekends)
    - Yearly seasonality (seasonal products)
    - Trend changepoints (growth/decline)

    TODO: m = Prophet(seasonality_mode='multiplicative')
          # multiplicative = seasonality scales with trend (realistic for growing businesses)
          # additive = seasonality is flat (use for stable, mature markets)
    TODO: if country_holidays: m.add_country_holidays(country_name=country_holidays)
    TODO: m.fit(df)
    TODO: return m
    """
    m = Prophet(
        seasonality_mode="multiplicative",
        yearly_seasonality=True,
        weekly_seasonality=True,
        daily_seasonality=False,
    )
    if country_holidays:
        m.add_country_holidays(country_name=country_holidays)
    m.fit(df)
    return m


def prophet_forecast(model, periods: int, freq: str = "W") -> pd.DataFrame:
    """
    Returns a DataFrame with columns: ds, yhat, yhat_lower, yhat_upper.

    yhat = point forecast
    yhat_lower / yhat_upper = 80% uncertainty interval

    Args:
        freq: 'D' = daily, 'W' = weekly, 'M' = monthly

    TODO: future = model.make_future_dataframe(periods=periods, freq=freq)
    TODO: forecast = model.predict(future)
    TODO: return forecast[['ds', 'yhat', 'yhat_lower', 'yhat_upper']].tail(periods)
    """
    future   = model.make_future_dataframe(periods=periods, freq=freq)
    forecast = model.predict(future)
    return forecast[["ds", "yhat", "yhat_lower", "yhat_upper"]].tail(periods)

1c. XGBoost for Tabular Demand Forecasting

When you have many features (promotions, price, weather, store type), XGBoost outperforms ARIMA and Prophet because it can learn non-linear relationships between features and demand.

import pandas as pd
import numpy as np
import xgboost as xgb
from sklearn.model_selection import TimeSeriesSplit
from sklearn.metrics import mean_absolute_percentage_error

def create_lag_features(df: pd.DataFrame, target_col: str, lags: list[int]) -> pd.DataFrame:
    """
    Creates lagged versions of the target column as features for XGBoost.

    Example: lags=[1, 7, 14] creates:
        demand_lag_1  = demand from 1 period ago
        demand_lag_7  = demand from 7 periods ago (last week)
        demand_lag_14 = demand from 14 periods ago (2 weeks ago)

    WHY: XGBoost is not a time-series model — it treats each row independently.
    By adding lagged values as features, you give it the temporal context it needs.
    This is called "supervised learning on time-series" and is the most common
    technique in Kaggle demand forecasting competitions.

    TODO: for lag in lags:
              df[f'{target_col}_lag_{lag}'] = df[target_col].shift(lag)
    TODO: df = df.dropna()   # remove rows where lags go before the series start
    TODO: return df
    """
    for lag in lags:
        df[f"{target_col}_lag_{lag}"] = df[target_col].shift(lag)
    return df.dropna()


def train_xgb_demand(X_train, y_train, X_val, y_val):
    """
    Trains an XGBoost regressor for demand forecasting.

    Key hyperparameters to tune:
    - n_estimators: number of trees (start at 500, tune with early stopping)
    - max_depth:    tree depth (3–6 for tabular data)
    - learning_rate: 0.01–0.1 (lower = more trees needed, often better)
    - subsample:    fraction of rows per tree (0.8 reduces overfitting)

    TODO: model = xgb.XGBRegressor(
              n_estimators=500,
              max_depth=5,
              learning_rate=0.05,
              subsample=0.8,
              early_stopping_rounds=50,
              eval_metric='mae',
          )
    TODO: model.fit(X_train, y_train, eval_set=[(X_val, y_val)], verbose=False)
    TODO: mape = mean_absolute_percentage_error(y_val, model.predict(X_val))
    TODO: return model, mape
    """
    model = xgb.XGBRegressor(
        n_estimators=500,
        max_depth=5,
        learning_rate=0.05,
        subsample=0.8,
        early_stopping_rounds=50,
        eval_metric="mae",
    )
    model.fit(X_train, y_train, eval_set=[(X_val, y_val)], verbose=False)
    mape = mean_absolute_percentage_error(y_val, model.predict(X_val))
    return model, mape

Pillar 2 — Inventory Optimization (Balancing the Scale)

The Business Problem: You know demand is roughly 500 units/week. How many units should you keep in your warehouse at all times? Too many = storage cost. Too few = stockout = lost revenue.

The Data Science Skill: Stochastic Modeling and Simulation

2a. Economic Order Quantity (EOQ)

The EOQ formula gives you the optimal batch size for a single product with deterministic demand. You know this from your IE degree — here it is in Python.

import numpy as np

def economic_order_quantity(
    annual_demand: float,
    order_cost: float,
    holding_cost_per_unit_year: float,
) -> dict:
    """
    Computes the EOQ and derived inventory policy metrics.

    Formula: EOQ = sqrt(2 * D * S / H)
        D = annual demand (units/year)
        S = ordering cost ($/order) — cost to place one purchase order
        H = holding cost ($/unit/year) — warehouse, insurance, capital cost

    Returns a dict with:
        eoq:                optimal order quantity
        orders_per_year:    how many times to reorder annually
        cycle_time_days:    average days between orders
        annual_total_cost:  ordering cost + holding cost (minimized at EOQ)

    Note: EOQ assumes deterministic demand (no randomness). In reality,
    demand is stochastic, which is why we add safety stock in 2b below.
    """
    eoq = np.sqrt((2 * annual_demand * order_cost) / holding_cost_per_unit_year)
    orders_per_year = annual_demand / eoq
    cycle_time_days = 365 / orders_per_year
    annual_total_cost = (
        (annual_demand / eoq) * order_cost       # ordering cost component
        + (eoq / 2) * holding_cost_per_unit_year  # holding cost component
    )
    return {
        "eoq": round(eoq),
        "orders_per_year": round(orders_per_year, 1),
        "cycle_time_days": round(cycle_time_days, 1),
        "annual_total_cost_usd": round(annual_total_cost, 2),
    }


# Example:
result = economic_order_quantity(
    annual_demand=5200,          # 100 units/week * 52 weeks
    order_cost=50.0,             # $50 to place each order
    holding_cost_per_unit_year=2.0,  # $2/unit/year to store
)
print(result)
# → {'eoq': 510, 'orders_per_year': 10.2, 'cycle_time_days': 35.8, 'annual_total_cost_usd': 1020.0}

2b. Safety Stock with Monte Carlo Simulation

EOQ assumes demand is constant. Reality: demand fluctuates randomly. Safety stock is the buffer inventory you keep to cover demand spikes during lead time.

import numpy as np

def simulate_safety_stock(
    demand_mean: float,
    demand_std: float,
    lead_time_days: int,
    service_level: float = 0.95,
    n_simulations: int = 100_000,
) -> dict:
    """
    Uses Monte Carlo simulation to find the safety stock level that achieves
    the target service level (probability of not stocking out).

    Why Monte Carlo instead of the analytical formula?
    The analytical formula (safety_stock = Z * std * sqrt(lead_time)) assumes
    demand is normally distributed. Real demand often has:
    - Heavy tails (occasional massive orders)
    - Seasonality
    - Autocorrelation (today's demand correlates with yesterday's)
    Monte Carlo makes NO distributional assumptions — you simulate thousands
    of scenarios from your actual historical demand distribution.

    TODO: simulate total demand during lead time for n_simulations scenarios:
          total_demands = np.array([
              np.sum(np.random.normal(demand_mean, demand_std, lead_time_days))
              for _ in range(n_simulations)
          ])

          Vectorized (faster):
          daily_demands = np.random.normal(demand_mean, demand_std,
                                           (n_simulations, lead_time_days))
          total_demands = daily_demands.sum(axis=1)

    TODO: safety_stock = np.percentile(total_demands, service_level * 100) - (demand_mean * lead_time_days)
    TODO: reorder_point = demand_mean * lead_time_days + safety_stock

    Returns:
        safety_stock: units to keep as buffer
        reorder_point: order when inventory drops to this level
        service_level_achieved: verify the simulation matches the target
    """
    rng = np.random.default_rng(seed=42)
    daily_demands = rng.normal(demand_mean, demand_std, (n_simulations, lead_time_days))
    daily_demands = np.maximum(0, daily_demands)   # demand cannot be negative
    total_demands = daily_demands.sum(axis=1)

    expected_demand_lt = demand_mean * lead_time_days
    safety_stock  = np.percentile(total_demands, service_level * 100) - expected_demand_lt
    reorder_point = expected_demand_lt + safety_stock

    # Verify: what fraction of simulations are covered by this safety stock?
    actual_service_level = np.mean(total_demands <= reorder_point)

    return {
        "safety_stock_units": round(safety_stock),
        "reorder_point_units": round(reorder_point),
        "expected_demand_during_lt": round(expected_demand_lt),
        "service_level_achieved": round(actual_service_level, 4),
    }


# Example:
result = simulate_safety_stock(
    demand_mean=100,      # 100 units/day average
    demand_std=20,        # std dev of 20 units/day
    lead_time_days=7,     # 7-day supplier lead time
    service_level=0.95,   # 95% in-stock probability
)
print(result)
# → {'safety_stock_units': 53, 'reorder_point_units': 753, ...}

NetworkX: Mapping the Supply Chain as a Graph

Before you optimize routes, you model the supply chain as a directed graph. NetworkX is the standard library for this.

# pip install networkx
import networkx as nx
import matplotlib.pyplot as plt

def build_supply_chain_graph(facilities: list[dict], routes: list[dict]) -> nx.DiGraph:
    """
    Builds a directed graph of the supply chain network.

    Args:
        facilities: list of dicts with keys: id, name, type (supplier/warehouse/demand)
        routes:     list of dicts with keys: from_id, to_id, cost, lead_time_days

    Why a directed graph?
    - Flow goes in one direction (supplier → warehouse → demand point)
    - A DiGraph preserves this directionality
    - NetworkX algorithms (shortest path, max flow) all work on DiGraph

    TODO: G = nx.DiGraph()
    TODO: for f in facilities: G.add_node(f['id'], **f)
    TODO: for r in routes:     G.add_edge(r['from_id'], r['to_id'], **r)
    TODO: return G
    """
    G = nx.DiGraph()
    for f in facilities:
        G.add_node(f["id"], label=f["name"], node_type=f["type"])
    for r in routes:
        G.add_edge(r["from_id"], r["to_id"],
                   cost=r["cost"], lead_time_days=r["lead_time_days"])
    return G


def find_cheapest_path(G: nx.DiGraph, source: str, target: str) -> dict:
    """
    Finds the lowest-cost path from source to target in the supply chain graph.

    Uses Dijkstra's algorithm — the same algorithm used in GPS navigation.
    Edge weight = cost per unit shipped along that route.

    TODO: path = nx.dijkstra_path(G, source, target, weight='cost')
    TODO: cost = nx.dijkstra_path_length(G, source, target, weight='cost')
    TODO: return {'path': path, 'total_cost': cost}
    """
    path = nx.dijkstra_path(G, source, target, weight="cost")
    cost = nx.dijkstra_path_length(G, source, target, weight="cost")
    return {"path": path, "total_cost_per_unit": cost}


def identify_bottleneck_nodes(G: nx.DiGraph) -> list[tuple]:
    """
    Returns nodes sorted by betweenness centrality — the nodes that most
    supply chain flow passes through.

    A high-centrality node is a single point of failure. If it goes down
    (port strike, natural disaster), the whole network is disrupted.
    This is how you do resilience analysis.

    betweenness_centrality(v) = fraction of all shortest paths that pass through v
    High value → this node is critical. Low value → easily bypassed.

    TODO: centrality = nx.betweenness_centrality(G, weight='cost')
    TODO: return sorted(centrality.items(), key=lambda x: x[1], reverse=True)
    """
    centrality = nx.betweenness_centrality(G, weight="cost")
    return sorted(centrality.items(), key=lambda x: x[1], reverse=True)

Chaining the Three Pillars Together

This is the full pipeline you build in projects/advanced/project-7-supply-chain/:

[Pillar 1] Demand Forecast
    Prophet predicts next month's demand per location
            ↓
[Pillar 2] Inventory Policy
    EOQ + Monte Carlo → how much to order, when to reorder
            ↓
[Pillar 3] Route Optimization
    OR-Tools VRP → optimal truck routes to fulfil the orders
            ↓
[Dashboard]
    Plotly map: nodes = facilities, edges = active routes
    Table: demand forecast vs. current stock vs. reorder point
    Financial summary: total logistics cost + stockout risk ($)

The key insight: each pillar feeds the next. The demand forecast tells the inventory optimizer how much to order. The inventory optimizer tells the route optimizer how many units to move and where. The dashboard translates all of it into dollars.

Interview Questions to Practise

QuestionWhat to say
"What's the difference between ARIMA and Prophet?""ARIMA is a statistical model that requires stationarity and manual tuning of p,d,q. Prophet handles multiple seasonalities, missing data, and holidays automatically. For retail demand with weekly + yearly patterns, Prophet is faster to deploy. For simple monthly series with few data points, ARIMA often wins on accuracy."
"How do you calculate safety stock?""The analytical formula assumes normal demand — Z σ √L. In production I use Monte Carlo simulation: simulate 100,000 lead time scenarios from the actual demand distribution, then find the inventory level that covers 95% of them. It makes no distributional assumptions and is easy to explain to non-technical managers."
"What is the VRP and how does it connect to demand forecasting?""The VRP finds optimal truck routes given a set of delivery demands. The demands come from the demand forecast. So the full pipeline is: forecast → inventory policy (how much to ship) → VRP (how to route the trucks). They're not separate problems — they're three sequential decisions."
"What KPIs do you track for supply chain performance?""OTIF (On-Time In-Full delivery rate), Inventory Turnover (Cost of Goods Sold / Avg Inventory), Days of Supply, Stockout Rate, and Total Logistics Cost as % of Revenue. When I build a dashboard, I show these alongside the model outputs so stakeholders see the business impact, not just model accuracy."
