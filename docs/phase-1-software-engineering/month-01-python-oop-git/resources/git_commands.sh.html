# ==============================================================================
# DAY 01 — WHAT IS GIT? SETUP
# ==============================================================================
#
# TEACHER SAYS:
# Git is a VERSION CONTROL SYSTEM — it tracks changes to your files over time.
# Think of it as "save game checkpoints" for your code.
# Without Git: "final_v2_FINAL_realfinal.py" — everyone has done this! 😅
# With Git: every change is tracked, you can go back to ANY previous state.
#
# GitHub is a WEBSITE that hosts your Git repositories online (cloud backup + sharing).
# Git = the tool.  GitHub = the platform.  They're different things!
#
# FIRST-TIME SETUP — run these once after installing Git:
# Replace with your actual name and email (same as your GitHub account):

git config --global user.name  "Your Name"
git config --global user.email "your@email.com"
git config --global core.editor "code --wait"     # Use VS Code as default editor
git config --global init.defaultBranch main       # Use 'main' as default branch name
git config --list                                  # Verify your settings

# ==============================================================================
# DAY 02 — CREATING YOUR FIRST REPOSITORY
# ==============================================================================
#
# TEACHER SAYS:
# A REPOSITORY (repo) is a folder that Git is tracking.
# git init turns any folder into a Git repository.
# git status is the command you'll use most — shows what's changed.
#
# THE 3 STAGES of Git:
#   Working Directory → Staging Area → Repository (commit history)
#
#   1. You EDIT files in the Working Directory
#   2. You STAGE changes with 'git add' (choosing what to save)
#   3. You COMMIT staged changes with 'git commit' (creating a checkpoint)
#
# git add = "I want to include this change in the next save"
# git commit = "Save this checkpoint with a description"

mkdir my_first_repo          # Create a new folder
cd my_first_repo             # Enter the folder

git init                     # Initialize Git — creates a hidden .git folder
git status                   # Shows: "No commits yet" — nothing tracked yet

# Create a file:
echo "# My First Project" > README.md
echo "print('Hello from my first repo!')" > main.py

git status                   # Now shows: Untracked files (red)

git add README.md            # Stage ONE file
git add main.py              # Stage another file
# OR stage everything at once:
git add .                    # Stage ALL changes in current folder

git status                   # Now shows: Changes to be committed (green)

git commit -m "Initial commit: add README and main.py"
# -m flag provides the commit MESSAGE — always write meaningful messages!

git log                      # See your commit history
git log --oneline            # Compact version — one line per commit

# ==============================================================================
# DAY 03 — BRANCHES
# ==============================================================================
#
# TEACHER SAYS:
# A BRANCH is an independent line of development.
# Think of it like a parallel universe — you make changes in a branch without
# affecting the main code. When you're happy, you MERGE the branch back.
#
# Why branches?
#   main (or master) → your stable, working code
#   feature branch   → try new things safely
#   bug-fix branch   → fix a bug without touching main
#
# Golden rule: NEVER commit directly to main for real projects.
# Always branch → develop → test → merge.
#
# EXAMPLE:

git branch                   # List all branches (* marks current branch)
git branch feature-login     # Create a new branch
git checkout feature-login   # Switch to the new branch
# OR create AND switch in one command (modern way):
git checkout -b feature-dashboard

# Now work on this branch — commits here DON'T affect main:
echo "# Dashboard code" > dashboard.py
git add .
git commit -m "feat: add dashboard skeleton"

git log --oneline            # See commits on this branch

# Switch back to main:
git checkout main
ls                           # dashboard.py is NOT here (it's on the other branch!)

# Merge your feature into main:
git checkout main
git merge feature-dashboard
ls                           # Now dashboard.py is here!

# Delete the merged branch (clean up):
git branch -d feature-dashboard

# ==============================================================================
# DAY 04 — GITHUB: REMOTE REPOSITORIES
# ==============================================================================
#
# TEACHER SAYS:
# REMOTE → your repo stored on GitHub (the "cloud backup")
# origin → the conventional name for your main remote
# push   → send your commits FROM local TO GitHub
# pull   → get latest commits FROM GitHub TO local
# clone  → download a repository from GitHub to your computer
#
# WORKFLOW FOR YOUR OWN PROJECT:
#   1. Create repo on GitHub (with no files — don't initialize!)
#   2. Connect your local repo to GitHub
#   3. Push your code
#
# WORKFLOW FOR COLLABORATING:
#   1. Clone the repo
#   2. Create a branch
#   3. Make changes
#   4. Push branch to GitHub
#   5. Open Pull Request (PR)
#   6. Get code reviewed
#   7. Merge PR
#
# EXAMPLE:

# Connect local repo to GitHub (replace URL with yours):
git remote add origin https://github.com/YOUR_USERNAME/your-repo-name.git
git remote -v                # Verify the connection

# Push to GitHub (first time):
git push -u origin main      # -u sets 'origin main' as default for future pushes
# After that, just use:
git push

# Download from GitHub:
git pull                     # Fetch + merge latest changes

# Clone a repo (download someone else's project):
git clone https://github.com/USERNAME/repo-name.git
cd repo-name
ls

# ==============================================================================
# DAY 05 — THE GITIGNORE FILE
# ==============================================================================
#
# TEACHER SAYS:
# .gitignore tells Git which files to NEVER track.
# Things you MUST NOT commit:
#   - passwords, API keys, .env files (security!)
#   - __pycache__, .pyc files (auto-generated — not needed)
#   - virtual environments (venv/, .venv/)
#   - OS files (.DS_Store, Thumbs.db)
#   - Build output, logs
#
# Create a .gitignore in your project root:

cat > .gitignore << 'EOF'
# Python
__pycache__/
*.py[cod]
*.pyo
*.pyd
.Python
*.egg
*.egg-info/
dist/
build/

# Virtual environments
venv/
.venv/
env/
.env

# Secrets & Config
.env
.env.local
*.secret
config_private.py

# Data files (often too large for Git)
*.csv
*.xlsx
data/raw/

# Jupyter
.ipynb_checkpoints/
*.ipynb

# IDE settings
.vscode/
.idea/
*.swp

# OS files
.DS_Store
Thumbs.db

# Logs
*.log
logs/
EOF

git add .gitignore
git commit -m "chore: add .gitignore"
git push

# ==============================================================================
# DAY 06 — FIXING MISTAKES
# ==============================================================================
#
# TEACHER SAYS:
# Everyone makes mistakes. Git makes them RECOVERABLE.
# The key commands for undoing:
#
#   git restore <file>       → discard changes in working dir (not yet staged)
#   git restore --staged <f> → unstage a file (undo git add)
#   git commit --amend       → fix your LAST commit (message or files)
#   git revert HEAD          → create a NEW commit that UNDOES the last one (safe!)
#   git reset --soft HEAD~1  → undo last commit but KEEP the changes staged
#   git reset --hard HEAD~1  → ⚠️  DANGER: undo last commit AND discard changes!
#
# RULE: Only use 'git reset --hard' on LOCAL, unpushed commits.
#       Never rewrite history on commits you've already pushed!
#
# EXAMPLE:

# Oops, staged wrong file:
git add wrong_file.py
git restore --staged wrong_file.py   # Unstage it

# Oops, made changes you want to discard:
echo "bad code" >> main.py
git restore main.py                  # Undo file changes (goes back to last commit)

# Oops, bad commit message:
git commit -m "typo in msesage"
git commit --amend -m "fix: correct typo in function name"   # Fix message

# Oops, committed something wrong, want to undo gracefully (safe for pushed commits):
git revert HEAD
# This creates a new "undo" commit — history preserved!

# See what a commit changed:
git show HEAD               # Details of latest commit
git show abc1234            # Details of a specific commit by hash

# DAY 07 — PROFESSIONAL WORKFLOW: PULL REQUESTS & CODE REVIEW
# ==============================================================================
#
# TEACHER SAYS:
# In professional teams, nobody pushes directly to main.
# The standard workflow is called GITHUB FLOW:
#   1. Pull latest main: git pull origin main
#   2. Create feature branch: git checkout -b feature/my-feature
#   3. Write code, commit frequently
#   4. Push branch: git push origin feature/my-feature
#   5. Open Pull Request (PR) on GitHub
#   6. Team reviews code, suggests changes
#   7. Make changes, push more commits to same branch
#   8. PR gets approved and merged
#   9. Delete the branch: git branch -d feature/my-feature
#
# COMMIT MESSAGE CONVENTION (Conventional Commits):
#   feat:     new feature
#   fix:      bug fix
#   docs:     documentation change
#   style:    formatting (no logic change)
#   refactor: code refactoring
#   test:     adding tests
#   chore:    build tools, dependencies
#
# EXAMPLE GOOD COMMIT MESSAGES:
#   feat: add student grade prediction model
#   fix: handle NaN values in data loader
#   docs: update README with installation steps
#   chore: add numpy and pandas to requirements.txt

# ==============================================================================
# CHEAT SHEET — MOST USED GIT COMMANDS
# ==============================================================================

# SETUP
git config --global user.name "Name"      # Set name
git config --global user.email "e@mail"  # Set email

# DAILY WORKFLOW
git status                               # What's changed?
git add .                                # Stage all changes
git add filename                         # Stage one file
git commit -m "message"                  # Commit with message
git push                                 # Push to GitHub
git pull                                 # Get latest from GitHub

# BRANCHES
git branch                               # List branches
git checkout -b branch-name              # Create and switch
git checkout branch-name                 # Switch branch
git merge branch-name                    # Merge into current
git branch -d branch-name                # Delete branch

# HISTORY & INSPECTION
git log --oneline                        # Compact history
git diff                                 # Unstaged changes
git diff --staged                        # Staged changes
git show HEAD                            # Latest commit details

# UNDO
git restore file                         # Discard working changes
git restore --staged file                # Unstage
git commit --amend                       # Fix last commit
git revert HEAD                          # Safe undo via new commit

# REMOTE
git remote add origin URL                # Connect to GitHub
git clone URL                            # Download a repo
git push -u origin main                  # First push
git fetch                                # Download changes (don't merge yet)
