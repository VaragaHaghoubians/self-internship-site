"""
CAPSTONE PROJECT — Cold-Chain Logistics Warehouse
===================================================
Scenario:
    A cold-chain warehouse stores medicine and temperature-sensitive food.
    A cooling system failure causes thousands of dollars in spoilage within hours.
    Your system: monitors temperature live, predicts failure before it happens,
    and automatically calculates the optimal inventory transfer plan.

This is your three IE superpowers in one deployable system:
    Superpower 1 (OR):    OR-Tools optimizer computes optimal transfer plan
    Superpower 2 (PM):    Survival analysis model predicts cooling failure
    Superpower 3 (EM):    Dashboard shows Financial Risk ($), not just a probability

Architecture:
    ESP32 Sensor → MQTT/HTTP → Django Backend → PostgreSQL
                                      ↓
                              Celery Worker (every hour)
                                      ↓
                           [ML Model] → failure risk score
                                      ↓ (if risk > threshold)
                           [OR-Tools] → optimal transfer plan
                                      ↓

class CoolingFailureModel:
    """
    Predicts the probability that the cooling system will fail within 48 hours.

    This is a survival analysis problem, NOT a binary classification problem.
    Binary classification would answer "will it fail? yes/no."
    Survival analysis answers "what is the probability of failure in the next T hours?"

    Why survival analysis?
    - You want a continuous risk score (0.0 to 1.0) for the Financial Risk panel.
    - You want to handle censored data — sensors that haven't failed yet are
      still informative (they tell you "it survived at least this long").
    - The Cox Proportional Hazards model interprets feature importance as
      "how much does a 1-unit increase in temp_slope multiply the failure rate?"
      This is directly explainable to a non-technical manager.

    Library: lifelines (pip install lifelines)
    Key class: lifelines.CoxPHFitter

    Fallback option (simpler, for prototyping):
    - Use scikit-learn IsolationForest as an anomaly detector.
      Map the anomaly score to a probability via min-max scaling.
      This is less statistically rigorous but faster to implement.
      Use it first, then upgrade to Cox when it works end-to-end.

    Model file: saved as 'cooling_failure_model.pkl' alongside this script.

@dataclass
class InventoryItem:
    """
    Represents one pallet or batch of inventory in the at-risk cooler.

    The relocation_costs dict maps destination warehouse IDs to the estimated
    cost of moving this item there (transport + spoilage risk during transit).
    """

    description: str            # e.g. "Insulin — 500 vials"
    quantity_units: int
    value_usd: float            # total value — used to compute spoilage risk
    relocation_costs: dict      # {"warehouse_B": 120.0, "warehouse_C": 180.0}


@dataclass
class DestinationWarehouse:
    """A warehouse that can receive inventory from the at-risk cooler."""
    warehouse_id: str
    name: str
    available_capacity_units: int   # how many units it can absorb
    distance_km: float              # from the at-risk warehouse


class InventoryOptimizer:
    """
    Given a list of inventory items and destination warehouses,
    computes the optimal assignment of items to warehouses to minimize
    total relocation cost while respecting capacity constraints.

    This is the Assignment Problem — a classic IE/OR problem.
    We solve it with Google OR-Tools CP-SAT solver.

    Why not just use scipy.optimize.linear_sum_assignment?
    Because that assumes one-to-one assignment (one item per warehouse).
    We need many-to-one assignment with capacity constraints — OR-Tools handles this.
    """

    def solve(
