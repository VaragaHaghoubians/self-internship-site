Prerequisites — Project 5: Agentic Factory Digital Twin

Role: AI Systems Engineer | Phase: Advanced (after Month 10–12) | Reference project for Month 9

This project is the most architecturally complex of the three advanced projects. It combines discrete-event simulation (a university-level IE topic) with real-time anomaly detection and an autonomous LLM agent. Do not start this until you have completed Phase 5 (Capstone) and begun Phase 6 (Generative AI).

What This Project Is

A running software simulation of a factory production line. Machines process items in sequence. Stochastic breakdowns occur. An anomaly detection model flags when a machine is bottlenecking the line. An LLM agent reasons about the problem and autonomously applies a corrective action — then the simulation continues.

The loop: observe → detect anomaly → reason → act → observe again.

Discrete-Event Simulation with SimPy

This is the most unfamiliar concept in the project. Read the SimPy documentation before writing any code.

Documentation: https://simpy.readthedocs.io/en/latest/

Core concepts you must understand

import simpy

env = simpy.Environment()   # the simulation clock

# A "process" is a Python generator function — it yields events to pause and resume
def machine_process(env, name, resource):
    while True:
        # Request the resource (wait in queue if it's busy)
        with resource.request() as req:
            yield req                           # suspend: wait until resource is free
            process_time = random.expovariate(1 / 5.0)  # draw from exponential distribution
            yield env.timeout(process_time)     # suspend: simulate processing time
            print(f"{name} finished at {env.now:.2f}")

machine = simpy.Resource(env, capacity=1)  # one machine, no parallel processing
env.process(machine_process(env, "Machine A", machine))
env.run(until=100)   # run simulation for 100 time units

Key insight: yield env.timeout(t) does NOT sleep for t seconds of real time. It advances the simulation clock by t. A 1,000-hour simulation runs in milliseconds.

What you must be able to build

  • simpy.Resource with request/release logic
  • Breakdown modelling: a separate SimPy process that interrupts the main one with env.process(breakdown()) and process.interrupt()
  • Event logging: append every event (machine start, finish, breakdown, repair) to a pandas DataFrame
  • Multiple machines in sequence: the output queue of Machine A feeds the input queue of Machine B

Stochastic Modelling (IE / Operations Research Background)

Your IE coursework covers this. The code just puts it into practice:

Always use a seed for reproducibility: np.random.seed(42) at the top of your script.

Anomaly Detection with IsolationForest

from sklearn.ensemble import IsolationForest
import numpy as np

# IsolationForest scores each data point
# Anomalies are points that are "isolated" quickly in random trees
clf = IsolationForest(contamination=0.05, random_state=42)  # 5% expected anomalies
clf.fit(normal_operation_data)

# Score new observations — negative score = more anomalous
scores = clf.decision_function(new_data)
is_anomaly = clf.predict(new_data) == -1   # -1 = anomaly, 1 = normal

What counts as a "feature" here? Each row in your feature matrix represents one time window (e.g., the last 10 simulation cycles). Features include:

  • Machine queue length (moving average)
  • Inter-cycle time (moving average + standard deviation)
  • Number of breakdowns in the window
  • Throughput rate (units/time)

LLM Tool Calling (The Agentic Part)

Before building the LLM agent, understand the ReAct pattern (Reason + Act):

Thought: "Queue at Machine 2 is growing. Throughput dropped 30% in the last 10 cycles."
Action: call tool `get_machine_stats(machine_id="M2")`
Observation: "M2 average process time: 8.2 min (baseline: 5.1 min)"
Thought: "M2 is slowing down — possible early breakdown. I should reduce its workload."
Action: call tool `reroute_workload(from_machine="M2", to_machine="M3", pct=40)`
Observation: "Rerouted 40% of M2 workload to M3. Queue decreasing."

Each tool is a Python function your agent can call:

import openai

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_machine_stats",
            "description": "Returns current queue length, average processing time, and breakdown count for a machine",
            "parameters": {
                "type": "object",
                "properties": {
                    "machine_id": {"type": "string", "description": "Machine identifier, e.g. 'M1'"}
                },
                "required": ["machine_id"],
            },
        },
    },
]

response = openai.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Machine M2 is showing anomalies. Diagnose and fix."}],
    tools=tools,
    tool_choice="auto",
)

Study before coding:

  • OpenAI function/tool calling documentation: https://platform.openai.com/docs/guides/function-calling
  • Understand the message loop: user → LLM → tool call → tool result → LLM → final answer

Python Generators

SimPy is built entirely on Python generators. You must understand yield deeply:

# A generator function — contains yield, returns a generator object
def countdown(n):
    while n > 0:
        yield n    # pause here, return n to caller
        n -= 1

# Consuming the generator
gen = countdown(5)
next(gen)  # 5
next(gen)  # 4
# ... SimPy's env.run() does this internally for all processes

If you do not understand generators, read this first: https://realpython.com/introduction-to-python-generators/

The Build Order — Do Not Skip Steps

  1. Step 1 (no AI): Get ProductionLine running and printing events to console. Verify the SimPy logic is correct by checking that queue lengths grow when a downstream machine is slow.
  2. Step 2 (no LLM): Add AnomalyMonitor using IsolationForest. Verify it flags obvious bottlenecks.
  3. Step 3 (rule-based): Add an OptimizerAgent that takes rule-based corrective actions (no LLM). This validates your tool functions work correctly.
  4. Step 4 (LLM): Replace the rule engine with an LLM that calls the same tool functions. The tools do not change — only the decision-maker changes.
  5. Step 5 (dashboard): Build the Streamlit dashboard last. It should just visualise data that already exists in your event log.

Building the dashboard first is the most common mistake. Build the engine first.
