Prerequisites — Project 6: Predictive Maintenance with Causal NLP

Role: Senior ML Engineer | Phase: Advanced (after Month 9) | Core reference for Capstone

This is the most directly useful advanced project for your job hunt. Every company with physical assets (factories, fleets, data centres, cold-chain warehouses) pays a premium for engineers who can build predictive maintenance systems. Study this project carefully even if you do not build it fully — the NASA CMAPSS dataset alone appears in interviews.

What This Project Is

Two-part system:

  1. ML regression model that predicts Remaining Useful Life (RUL) — "this engine will fail in ~42 more cycles"
  2. RAG + LLM pipeline that retrieves similar historical maintenance notes and generates a diagnostic report explaining why the failure is likely

Time-Series Feature Engineering — The Core Skill

Raw sensor data is just a sequence of numbers. The model cannot learn from raw readings alone — you must create window-based features that capture how the sensor data is changing over time.

import pandas as pd
import numpy as np

def add_rolling_features(df: pd.DataFrame, sensor_cols: list, window: int = 10) -> pd.DataFrame:
    """
    For each sensor, compute rolling statistics over the last `window` cycles.
    Rolling features are how you give a regression model 'memory' of the past.
    """
    df = df.sort_values(["unit_id", "cycle"])
    grouped = df.groupby("unit_id")

    for col in sensor_cols:
        # Rolling mean — is the sensor trending up or down?
        df[f"{col}_rolling_mean"] = grouped[col].transform(
            lambda x: x.rolling(window, min_periods=1).mean()
        )
        # Rolling std — how much is the sensor fluctuating?
        df[f"{col}_rolling_std"] = grouped[col].transform(
            lambda x: x.rolling(window, min_periods=1).std().fillna(0)
        )
        # Difference from previous cycle — rate of change
        df[f"{col}_diff"] = grouped[col].transform(lambda x: x.diff().fillna(0))

    return df

def compute_rul(df: pd.DataFrame) -> pd.DataFrame:
    """
    Add the target variable: Remaining Useful Life.
    RUL = (max cycle for this engine) - (current cycle)
    So at cycle 1, RUL is high. At the last cycle before failure, RUL = 0.
    """
    max_cycle = df.groupby("unit_id")["cycle"].transform("max")
    df["RUL"] = max_cycle - df["cycle"]
    return df

Why clip RUL? The early cycles of every engine are healthy and similar — they are not informative for predicting failure. Clipping RUL at 125 (a common choice for CMAPSS FD001) means the model focuses only on the degradation zone.

df["RUL"] = df["RUL"].clip(upper=125)

Variance-Based Feature Selection

CMAPSS has 21 sensor columns, but many sensors show almost zero variance across the entire dataset — they are constant and carry no information about degradation. Drop them before modelling.

from sklearn.feature_selection import VarianceThreshold

selector = VarianceThreshold(threshold=0.01)
X_selected = selector.fit_transform(X)
kept_features = X.columns[selector.get_support()].tolist()
print(f"Kept {len(kept_features)} of {X.shape[1]} features")

Typically 7–9 sensors survive this filter for FD001.

The NASA Custom Scoring Function — Do Not Skip This

The standard RMSE does not capture the real cost of predictive maintenance predictions. A late prediction (you said 10 cycles remaining but the engine failed in 2) is far more dangerous than an early prediction (you said 10 cycles but it actually had 20 left — unnecessarily early maintenance is just money, not catastrophe).

import numpy as np

def nasa_scoring_function(y_true: np.ndarray, y_pred: np.ndarray) -> float:
    """
    The official NASA CMAPSS evaluation metric.
    Penalises late predictions exponentially more than early predictions.

    d < 0: early prediction (predicted RUL > true RUL) → penalty = exp(-d/13) - 1
    d >= 0: late prediction  (predicted RUL < true RUL) → penalty = exp(d/10) - 1
    """
    d = y_pred - y_true
    scores = np.where(d < 0, np.exp(-d / 13) - 1, np.exp(d / 10) - 1)
    return float(np.sum(scores))

# Lower score is better. A perfect predictor scores 0.
score = nasa_scoring_function(y_test, preds)
print(f"NASA Score: {score:.1f}  (lower is better, 0 is perfect)")

In interviews: If someone asks about your evaluation metric, saying "I used RMSE" is average. Saying "I implemented the NASA asymmetric scoring function because late predictions carry a disproportionate cost in real maintenance planning" gets you hired.

Optuna — Hyperparameter Tuning

Instead of manually trying different learning_rate and max_depth values, Optuna searches the parameter space automatically and finds the optimal combination.

import optuna
optuna.logging.set_verbosity(optuna.logging.WARNING)

def objective(trial):
    params = {
        "n_estimators":      trial.suggest_int("n_estimators", 100, 500),
        "learning_rate":     trial.suggest_float("learning_rate", 0.01, 0.3, log=True),
        "max_depth":         trial.suggest_int("max_depth", 3, 8),
        "subsample":         trial.suggest_float("subsample", 0.6, 1.0),
        "colsample_bytree":  trial.suggest_float("colsample_bytree", 0.6, 1.0),
    }
    model = xgb.XGBRegressor(**params, random_state=42)
    model.fit(X_train, y_train, eval_set=[(X_val, y_val)], verbose=False)
    preds = model.predict(X_val)
    return np.sqrt(mean_squared_error(y_val, preds))

study = optuna.create_study(direction="minimize")
study.optimize(objective, n_trials=50)
print("Best params:", study.best_params)

RAG Pipeline — Retrieval-Augmented Generation

The second half of the project retrieves similar historical technician notes to help the LLM generate a diagnostic report. This is the same RAG architecture used in Month 11.

What you need to understand

  1. Text embeddings: Convert a maintenance note ("replaced bearing, high vibration observed") into a 384-dimensional vector using sentence-transformers. Similar notes will have similar vectors.
  2. Vector database (ChromaDB): Store all historical note vectors. At query time, find the most similar ones using cosine similarity.
  3. LLM synthesis: Pass the retrieved notes + current sensor context to an LLM and ask it to generate a diagnostic report.
from sentence_transformers import SentenceTransformer
import chromadb

# Step 1: embed historical notes
embed_model = SentenceTransformer("all-MiniLM-L6-v2")   # fast, small, good quality
notes = ["bearing replacement after high vibration", "coolant leak detected near pump 3", ...]
embeddings = embed_model.encode(notes)

# Step 2: store in ChromaDB
client = chromadb.Client()
collection = client.create_collection("maintenance_notes")
collection.add(
    documents=notes,
    embeddings=embeddings.tolist(),
    ids=[str(i) for i in range(len(notes))],
)

# Step 3: retrieve similar notes for a new observation
query = "sensor s4 spike, temperature exceeding threshold"
query_embedding = embed_model.encode([query])
results = collection.query(query_embeddings=query_embedding.tolist(), n_results=3)
retrieved_notes = results["documents"][0]

The CMAPSS Dataset — What to Expect

The raw files are space-separated .txt files with no header row. The column layout is:

unit_id | cycle | op1 | op2 | op3 | s1 | s2 | ... | s21

Start with FD001.txt only — one operating condition, one fault mode. It is the cleanest subset. FD002 and FD004 (multiple operating conditions) require normalisation within each operating condition, which adds complexity.
