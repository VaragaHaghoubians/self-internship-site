Prerequisites — Project 3: House Price Prediction Pipeline

Role: ML Engineer | Phase: 2 | Suggested timing: After Month 3

This project is not about getting the best RMSE. It is about building a pipeline that an engineer can maintain — one that handles preprocessing automatically, can be retrained in one command, and can be saved and loaded without re-running all the training code. The engineering discipline matters as much as the modelling here.

pandas & numpy — Feature Engineering Focus

You already know the basics from Projects 1 and 2. For this project you need to go deeper:

import pandas as pd
import numpy as np

# Log transformation — critical for skewed targets like house prices
df["SalePrice_log"] = np.log1p(df["SalePrice"])
# Use log1p (log(1+x)) instead of log(x) because it handles zeros safely

# Reversing the transformation after prediction
predictions_actual = np.expm1(model.predict(X_test))  # inverse of log1p

# Creating interaction features
df["TotalSF"] = df["TotalBsmtSF"] + df["1stFlrSF"] + df["2ndFlrSF"]
df["HouseAge"] = df["YrSold"] - df["YearBuilt"]
df["Remodeled"] = (df["YearRemodAdd"] != df["YearBuilt"]).astype(int)

# Counting missing values per column
df.isnull().sum().sort_values(ascending=False)

Critical concept — why log-transform the target? House prices are right-skewed (a few mansions inflate the mean). Linear regression assumes normally distributed errors. Log-transforming the target makes the distribution approximately normal, which is why almost every Kaggle winner on this dataset uses it.

scikit-learn — Pipelines (The Core Concept)

This is the most important skill in this project. A Pipeline chains preprocessing steps and a model into a single object that you can fit, predict, and save as one unit.

Why Pipelines matter

Without a Pipeline, you have to remember to apply the same transformations to training data, validation data, AND new predictions — separately, in the right order. One mistake means data leakage or wrong predictions. A Pipeline makes this automatic and impossible to get wrong.

from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OrdinalEncoder
from sklearn.impute import SimpleImputer
from sklearn.ensemble import RandomForestRegressor

# Separate numeric and categorical columns
numeric_features = ["LotArea", "OverallQual", "YearBuilt", "GrLivArea"]
categorical_features = ["Neighborhood", "HouseStyle", "SaleCondition"]

# Build preprocessing sub-pipeline for numeric columns
numeric_transformer = Pipeline(steps=[
    ("imputer", SimpleImputer(strategy="median")),  # fill nulls with median
    ("scaler", StandardScaler()),                    # normalise
])

# Build preprocessing sub-pipeline for categorical columns
categorical_transformer = Pipeline(steps=[
    ("imputer", SimpleImputer(strategy="most_frequent")),  # fill nulls with mode
    ("encoder", OrdinalEncoder(handle_unknown="use_encoded_value", unknown_value=-1)),
])

# Combine both with ColumnTransformer
preprocessor = ColumnTransformer(transformers=[
    ("num", numeric_transformer, numeric_features),
    ("cat", categorical_transformer, categorical_features),
])

# Add a model at the end → this is the full pipeline
full_pipeline = Pipeline(steps=[
    ("preprocessor", preprocessor),
    ("regressor", RandomForestRegressor(n_estimators=200, random_state=42)),
])

# Fit → predict → save — one object does all three
full_pipeline.fit(X_train, y_train)
preds = full_pipeline.predict(X_test)

import joblib
joblib.dump(full_pipeline, "models/house_price_pipeline.pkl")

Key classes to understand

  • SimpleImputer — fills missing values with mean, median, or most frequent
  • StandardScaler — scales features to zero mean, unit variance
  • OrdinalEncoder — converts text categories to integers
  • OneHotEncoder — converts text categories to binary columns (for unordered categories)
  • ColumnTransformer — applies different transformers to different subsets of columns

Regression Models

You will compare at least three models. Know what each one does before tuning it.

ModelKey ideaHyperparameters to tune
Linear RegressionFits a straight-line relationship; interpretable baselineNone (or use Ridge/Lasso for regularisation)
RidgeLinear Regression + L2 regularisation — shrinks coefficients to reduce overfittingalpha (regularisation strength)
Random Forest RegressorEnsemble of decision trees; handles non-linearity; robustn_estimators, max_depth, min_samples_split

Start with Ridge (fast, interpretable), then use XGBoost for final performance.

Model Evaluation for Regression

MetricFormulaWhat it means
RMSEsqrt(mean((pred - true)²))Error in the same units as the target (dollars). Penalises large errors more.
MAEmean(abs(pred - true))Average error in dollars. Easier to explain to business.
R²1 - SS_res / SS_totHow much variance is explained? 0.90 means the model explains 90% of price variation.
from sklearn.metrics import mean_squared_error, mean_absolute_error, r2_score
import numpy as np

rmse = np.sqrt(mean_squared_error(y_test, preds))
mae  = mean_absolute_error(y_test, preds)
r2   = r2_score(y_test, preds)
print(f"RMSE: ${rmse:,.0f}  |  MAE: ${mae:,.0f}  |  R²: {r2:.4f}")

Cross-validation — always evaluate with cross-validation, not just one train/test split:

from sklearn.model_selection import cross_val_score
scores = cross_val_score(pipeline, X, y, cv=5, scoring="neg_root_mean_squared_error")
print(f"CV RMSE: {-scores.mean():,.0f} ± {scores.std():,.0f}")

Missing Value Handling — This Dataset Has a Lot

The Ames Housing dataset has ~80 features, many with missing values that are not random — they have meaning.

Example: GarageType is null because the house has no garage, not because data is missing. You must read data_description.txt to understand which nulls mean "none" and which mean "unknown."

PatternHow to handle it
Null means "None" (no garage, no pool)Fill with "None" (string) or 0 (numeric)
Null is truly missingFill with median (numeric) or most frequent (categorical)
Column has >80% nullsDrop the column entirely

Saving and Loading the Pipeline

import joblib

# Save after training
joblib.dump(full_pipeline, "models/house_price_pipeline.pkl")

# Load in a new script or API endpoint — no retraining needed
loaded_pipeline = joblib.load("models/house_price_pipeline.pkl")
new_prediction = loaded_pipeline.predict(new_house_data)

Why this matters: In a production system, you train once and deploy the saved file. Every time a user submits a house for valuation, you load the file and call .predict(). You never retrain on every request.
